# Bitwrench Versioning & Release Strategy

**Status**: Authoritative design document (March 2026)

## Version Chain

Single source of truth: `package.json`

```
package.json
  └─→ tools/generate-version.cjs  (prebuild hook)
       └─→ src/version.js         (auto-generated, exports VERSION + VERSION_INFO)
            └─→ rollup             (bundles into all dist formats)
                 └─→ dist/*.js     (banner: /*! bitwrench vX.Y.Z | license | homepage */)
                      └─→ bw.getVersion()  (runtime access)
```

CI enforces: `bw.version === package.json.version` (test/bitwrench_ci.js).

## Directory Layout

```
dist/                  ← v2 builds only (shipped to npm via "files" field)
  bitwrench.umd.js     bitwrench.umd.min.js
  bitwrench.esm.js     bitwrench.esm.min.js
  bitwrench.cjs.js     bitwrench.cjs.min.js
  bitwrench.es5.js     bitwrench.es5.min.js
  bitwrench.css         builds.json
  *.map

releases/v1/           ← frozen v1.2.16 archive (NOT shipped to npm)
  bitwrench.esm.js     bitwrench.esm.min.js
  bitwrench.umd.min.js bitwrench.css
  *.sri.txt             VERSION.txt  README.md

releases/v2/           ← created by `npm run build:release` (NOT shipped to npm)
  (copies of dist/*.js and dist/*.css, no maps)
  VERSION.txt
```

## npm Publish Strategy

The `"files"` field in package.json controls what ships to npm:

```json
"files": ["dist/", "src/", "README.md", "LICENSE.txt"]
```

- `dist/` contains only v2 builds
- `releases/` is NOT shipped — it's for browsing on GitHub only
- Source maps are included in the npm package (inside `dist/`)

### dist-tags

| Tag | Version | Description |
|-----|---------|-------------|
| `latest` | 2.x | Default install target |
| `v1` | 1.2.16 | Legacy, for existing v1 users: `npm install bitwrench@v1` |

To publish with the v1 tag (one-time, from the frozen v1 release):
```bash
npm dist-tag add bitwrench@1.2.16 v1
```

## CDN Strategy

Both unpkg and jsdelivr resolve from the npm tarball:

```
https://unpkg.com/bitwrench                    → dist/bitwrench.umd.js (via "main")
https://unpkg.com/bitwrench/dist/bitwrench.esm.js
https://cdn.jsdelivr.net/npm/bitwrench         → dist/bitwrench.umd.js
https://cdn.jsdelivr.net/npm/bitwrench@1.2.16  → v1 frozen release
```

No special CDN setup needed — the `"main"`, `"module"`, and `"exports"` fields handle resolution.

## GitHub Pages

GitHub Pages serves the entire repo (or docs/ if configured). The `releases/` directory is browsable there for users who want direct file access without npm.

- `index.html` is built with bitwrench (dog-fooding)
- Example pages served from root
- API reference generated by `tools/build-api-reference.js`

## Git Tag Convention

Tags follow: `vX.Y.Z` (e.g., `v2.0.4`)

```bash
git tag v2.0.4
git push --tags
```

Tags are created manually as part of the release process, not automated.

## SRI (Subresource Integrity)

**Status**: Needs reimplementation for v2.

The v1 SRI generation (`generate-sri` script) hashed root-level files that no longer exist. A new implementation should:

1. Hash each file in `dist/` using SHA-384
2. Output to `dist/sri.json` (machine-readable) rather than individual `.txt` files
3. Optionally print integrity attributes for `<script>` tags

This is tracked as a separate future task.
