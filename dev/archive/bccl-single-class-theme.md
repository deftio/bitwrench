# BCCL Single-Class Theme Architecture

## Problem

The current themed CSS system generates ~140-180 selectors per theme via 32
`generate*Themed()` functions and two utility generators. Each component type
gets its own color-variant selectors (`.bw_btn_primary`, `.bw_alert_danger`,
`.bw_card_accent.bw_border_success`, etc.). This creates:

- 11 variant forEach loops (now replaced by utility generators, but still ~126 rules)
- Per-component interaction rules (hover/active/focus) with compound selectors
- A `VARIANT_CLASSES` mapping layer in JS to translate variant names to utility classes
- ~1.9KB gzip bundle growth from the utility refactor

## Insight

A BCCL component is a TACO object. The TACO carries structural intent
via class names. CSS provides the visual layer. The current system over-
specifies the CSS side — generating per-component-per-variant selectors
when a single palette class on the root element would suffice.

Desktop UI frameworks (MFC, Swing, VCL) solved this decades ago: the
look-and-feel system applies to all components uniformly. You don't
have "button-primary-blue" and "card-primary-blue" as separate skins.
You have "primary = blue" and every component that says "I'm primary"
gets blue.

## New Architecture

### Layer 1: Reference TACO (component template)

Each BCCL component has a reference TACO stored in the registry.
`bw.make('card', props)` clones the template and fills in props.
The TACO has structural class names only:

```javascript
bw.makeCard({ title: 'Hello', variant: 'primary' })
// returns:
{
  t: 'div',
  a: { class: 'bw_card bw_primary' },
  //          ^^^^^^^^ ^^^^^^^^^^
  //          structural  theme (ONE class)
  c: [
    { t: 'div', a: { class: 'bw_card_body' }, c: [
      { t: 'h5', a: { class: 'bw_card_title' }, c: 'Hello' }
    ]}
  ]
}
```

### Layer 2: Structural CSS (unchanged)

`structuralRules` in bitwrench-styles.js. ~300 selectors across 44
categories. Layout, display, flex, padding, border-radius, sizing.
No colors. Already clean (3 minor leaks to fix).

```css
.bw-card { display: flex; flex-direction: column; border: 1px solid; border-radius: 0.25rem; }
.bw-card-body { flex: 1 1 auto; padding: 1rem; }
.bw-card-title { margin-bottom: 0.75rem; }
```

### Layer 3: Theme CSS (9 palette classes)

Generated by `generateThemedCSS()`. Instead of per-component-per-variant
selectors, we generate 9 palette classes:

```css
/* One rule per palette color — applies to ANY component */
.bw-primary   { color: #fff; background-color: #0d6efd; border-color: #0d6efd; }
.bw-secondary { color: #fff; background-color: #6c757d; border-color: #6c757d; }
.bw-success   { color: #fff; background-color: #198754; border-color: #198754; }
/* ... 6 more */
```

Plus pseudo-state rules (shared, not per-component):

```css
.bw-primary:hover   { background-color: #0b5ed7; border-color: #0a58ca; }
.bw-primary:active  { background-color: #0a58ca; }
.bw-primary:focus-visible { box-shadow: 0 0 0 3px rgba(13,110,253,0.25); outline: none; }
```

Plus a few component-specific overrides where the palette class means
something slightly different (e.g. alerts use light bg + dark text):

```css
.bw-alert.bw-primary { background-color: #cfe2ff; color: #084298; border-color: #b6d4fe; }
.bw-toast.bw-primary { background-color: inherit; border-left: 4px solid #0d6efd; }
```

### Total selector count comparison

| Layer | Before | After |
|-------|--------|-------|
| Structural | ~300 | ~315 (absorb ~15 layout rules from themed) |
| Themed: palette classes | 0 | 9 |
| Themed: pseudo-states | ~126 (interaction rules) | ~27 (3 states x 9 colors) |
| Themed: component overrides | ~140 (per-component variants) | ~30 (alert, toast, etc.) |
| Themed: utility generators | ~63 | 0 (palette classes replace these) |
| **Total themed** | **~329** | **~66** |

### What gets deleted from JS

1. `generatePaletteUtilities()` — replaced by 9 palette classes
2. `generateInteractionRules()` — replaced by shared pseudo-state rules
3. `VARIANT_CLASSES` mapping — no longer needed
4. 21 pure-color generate*Themed functions — collapsed into palette classes
5. 4 no-op functions — deleted entirely

### What changes in BCCL components

Components that currently emit utility classes:
```javascript
// CURRENT (utility class approach):
class: 'bw_btn bw_bg_primary bw_text_on_primary bw_border_primary'

// NEW (single class):
class: 'bw_btn bw_primary'
```

The `variant` prop just adds one class to the root element.
No mapping needed. No VARIANT_CLASSES lookup.

### Component-specific theme overrides

Some components interpret the palette class differently:

| Component | Default behavior | Override |
|-----------|-----------------|----------|
| button | bg=base, text=textOn | (default, no override) |
| badge | bg=base, text=textOn | (default, no override) |
| alert | bg=light, text=darkText, border=subtle | `.bw-alert.bw-primary { bg: light; color: dark; }` |
| toast | inherit bg, left border accent | `.bw-toast.bw-primary { border-left: 4px solid base; }` |
| stat card | inherit bg, left border accent | `.bw-stat-card.bw-primary { border-left-color: base; }` |
| timeline marker | bg=base (dot) | `.bw-timeline-marker.bw-primary { ... }` |
| spinner | text color only | `.bw-spinner-border.bw-primary { color: base; bg: transparent; }` |
| card (accent) | left border accent | `.bw-card.bw-primary { border-left: 4px solid base; }` |

This is ~8 component types with overrides x 9 colors = ~72 override rules.
But many can be collapsed (e.g. all "left border accent" components share
the same rule pattern).

### Scoped themes

Scoped themes still work. The scope class wraps everything:

```css
.ocean .bw-primary { background-color: #2196F3; ... }
.ocean .bw-primary:hover { background-color: #1976D2; ... }
```

### What stays the same

- `bw.generateTheme(name, config)` API — same inputs/outputs
- `bw.applyTheme()` / `bw.toggleTheme()` — same behavior
- `structuralRules` — unchanged (minus 3 color leaks to fix)
- All 47 make* functions — same API, just simpler class output
- `bw.make(type, props)` factory — same behavior

## Implementation Steps

1. Move ~15 layout rules from themed functions to structuralRules
2. Fix 3 cosmetic leaks in structuralRules (progress, spinner, formSwitch)
3. Write `generatePaletteClasses(scope, palette)` — 9 palette root classes
4. Write `generatePseudoStates(scope, palette)` — hover/active/focus per palette
5. Write `generateComponentOverrides(scope, palette)` — alert/toast/etc. overrides
6. Replace `generateThemedCSS()` orchestrator with new 3-function pipeline
7. Delete all old generate*Themed functions
8. Delete `generatePaletteUtilities()`, `generateInteractionRules()`, `VARIANT_CLASSES`
9. Simplify all BCCL components: variant → single class on root
10. Update tests
11. Build, verify gzip, visual check
