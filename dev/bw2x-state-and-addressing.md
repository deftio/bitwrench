# Bitwrench 2.x: State Management & Mutable Addressing

**Date**: March 2026
**Status**: Design (pre-implementation)
**Depends on**: `bw.uuid(prefix)`, `bw.DOM()` state preservation, `o.state` / `el._bw_state`

---

## 1. Problem Statement

Bitwrench currently has one update strategy: **nuke and rebuild**. When component state changes, `render(el)` calls `bw.DOM(el, newTaco)`, which:

1. Calls `bw.cleanup(el)` (fires unmount callbacks, removes `_bw_state` from children)
2. Sets `el.innerHTML = ''` (destroys all children)
3. Calls `bw.createDOM(newTaco)` (recreates entire subtree)
4. Appends the new subtree to `el`

This works. It's simple. But it has costs:

- **Performance**: O(n) DOM operations for an O(1) state change (e.g., changing "42" to "43")
- **Lost transient state**: Input focus, scroll position, text selection, CSS transition progress — all destroyed on re-render
- **Boilerplate**: Every onclick handler must manually call `render(el)`. Forget it once → stale UI.
- **No targeted updates**: Cannot change one value without rebuilding the entire component tree
- **Cross-component coupling**: Local buttons must know about external render functions (e.g., `renderGlobal()`) to keep coordination panels in sync

This document specifies three new capabilities to address these issues:

1. **`o.render`** — first-class render function (eliminates `mounted` boilerplate)
2. **`bw.update(el)`** — trigger re-render (eliminates manual `render(el)` calls)
3. **`bw.patch()`** — targeted DOM updates via UUID addressing (eliminates full rebuilds for value changes)

## 2. Addressing Model

### 2.1 The Question

Given a component hierarchy like:

```
div#bw_ctr_abc
  div.ctr-card
    h4 "Visitors"
    div.ctr-value "42"        ← how do we address this?
    div.ctr-btns
      button "+"
      button "-"
```

When the count changes from 42 to 43, how does bitwrench find and update `div.ctr-value` without rebuilding everything?

### 2.2 Three Strategies Evaluated

#### A. Relative Path (rejected)

Address by tree position: `el.children[0].children[1].textContent`

- **Fragile**: Adding a wrapper div breaks all paths
- **Not serializable**: Paths are meaningless outside the current DOM structure
- **How React avoids this**: vDOM diffing compares trees, doesn't use fixed paths

**Verdict**: Not viable.

#### B. UUID Addressing (chosen)

Each mutable element gets a stable UUID: `id="bw_val_f7a92b..."`

- **Stable**: ID stays the same even if the element moves within a list
- **Works with both code paths**: TACO→HTML (string) and TACO→DOM (nodes) both produce elements with IDs
- **Inspectable**: `document.querySelectorAll('[id^="bw_val_"]')` shows all bitwrench-managed values
- **Serializable**: UUIDs can be included in JSON from a server — client knows which elements are mutable
- **Cost**: More IDs in the DOM. Mitigated by the `bw_` prefix convention (easy to identify, easy to clean up)

**Verdict**: Best fit for bitwrench's data-first philosophy.

#### C. Direct Node Reference (considered, deferred)

Render closure captures the actual DOM node:

```javascript
var valNode = document.createElement('div');
// later: valNode.textContent = '43';
```

- **Fastest**: No DOM lookup at all
- **Fragile**: References go stale if any ancestor calls `bw.DOM()` (which destroys and recreates children)
- **Not serializable**: Node references can't cross a network boundary
- **How Svelte uses this**: Svelte controls the entire render pipeline, so it can guarantee references stay valid. Bitwrench can't — users can call `bw.DOM(parent, ...)` at any level.

**Verdict**: Could be an optimization layer later, but UUID addressing is the correct foundation. If we add this, it would be internal to `bw.patch()`, not user-facing.

### 2.3 UUID Convention

All bitwrench-generated IDs follow the pattern:

```
bw_{type}_{hex}
```

Examples:
- `bw_ctr_f7a92b3c...` — a counter component
- `bw_val_a1b2c3d4...` — a mutable value inside a component
- `bw_item_e5f6a7b8...` — a list item
- `bw_todo_11223344...` — a todo component

CSS selectability:
- `[id^="bw_"]` — all bitwrench-managed elements
- `[id^="bw_val_"]` — all mutable values
- `[id^="bw_ctr_"]` — all counter components

Generated by `bw.uuid(prefix)`:
- `bw.uuid()` → `bw_f7a92b3c...`
- `bw.uuid('val')` → `bw_val_f7a92b3c...`
- `bw.uuid('item')` → `bw_item_f7a92b3c...`

## 3. API Design

### 3.1 `o.render` — First-Class Render Function

**Current pattern** (boilerplate):

```javascript
function makeCounter(opts) {
  function render(el) {
    var s = el._bw_state;
    bw.DOM(el, { t: 'div', c: String(s.count) });
  }

  return {
    t: 'div', a: { id: bw.uuid('ctr') },
    o: {
      state: { count: 0 },
      mounted: function(el) { render(el); }  // boilerplate
    }
  };
}
```

**New pattern**:

```javascript
function makeCounter(opts) {
  return {
    t: 'div', a: { id: bw.uuid('ctr') },
    o: {
      state: { count: 0 },
      render: function(el) {                  // bitwrench calls this on mount
        var s = el._bw_state;
        bw.DOM(el, { t: 'div', c: String(s.count) });
      }
    }
  };
}
```

**Behavior**:
- If `o.render` is defined, bitwrench stores it as `el._bw_render` during `createDOM()`
- `o.render(el)` is called once after mount (replaces `o.mounted`)
- `o.render` and `o.mounted` are mutually exclusive. If both are provided, `o.render` wins (with console warning in dev mode).
- `el._bw_render` is preserved across `bw.DOM(el, ...)` re-renders (same as `el._bw_state`)

**Implementation in `bw.createDOM()`**:

```javascript
// In the lifecycle hooks section (~line 337):
if (opts.render) {
  el._bw_render = opts.render;
  // Queue initial render (same timing as mounted)
  requestAnimationFrame(() => {
    if (document.body.contains(el)) {
      opts.render(el, el._bw_state || {});
    }
  });
}
```

**In `bw.DOM()`** — preserve `_bw_render` alongside `_bw_state`:

```javascript
const savedState = targetEl._bw_state;
const savedRender = targetEl._bw_render;
const savedBwId = targetEl.getAttribute('data-bw-id');

bw.cleanup(targetEl);

if (savedState !== undefined) targetEl._bw_state = savedState;
if (savedRender) targetEl._bw_render = savedRender;
if (savedBwId) targetEl.setAttribute('data-bw-id', savedBwId);
```

### 3.2 `bw.update(el)` — Trigger Re-Render

**Purpose**: Re-run a component's render function. Replaces manual `render(el)` calls.

```javascript
bw.update = function(target) {
  var el = typeof target === 'string' ? document.querySelector(target) : target;
  if (el && el._bw_render) {
    el._bw_render(el, el._bw_state || {});
  }
  return el;
};
```

**Usage before** (manual render calls in every handler):

```javascript
onclick: function() { s.count++; render(el); }
```

**Usage after**:

```javascript
onclick: function() { s.count++; bw.update(el); }
```

**Benefits**:
- `render` is no longer a closure the user must wire up — it's stored on the element
- Works from anywhere: `bw.update(document.getElementById('bw_ctr_abc'))` re-renders that component from external code
- Enables cross-component coordination without coupling (see Section 5)

### 3.3 `bw.patch()` — Targeted DOM Updates

**Purpose**: Update specific elements by UUID without rebuilding the component tree.

#### 3.3.1 Single-value patch

```javascript
bw.patch = function(id, content, attr) {
  var el = typeof id === 'string' ? document.getElementById(id) : id;
  if (!el) return null;

  if (attr) {
    // Patch an attribute
    el.setAttribute(attr, String(content));
  } else if (typeof content === 'object' && content !== null && content.t) {
    // Patch with a TACO — replace children
    el.innerHTML = '';
    el.appendChild(bw.createDOM(content));
  } else {
    // Patch text content
    el.textContent = String(content);
  }
  return el;
};
```

#### 3.3.2 Multi-value patch (batch)

```javascript
bw.patchAll = function(patches) {
  // patches: { 'bw_val_abc': '43', 'bw_val_def': 'hello', ... }
  var results = {};
  for (var id in patches) {
    if (patches.hasOwnProperty(id)) {
      results[id] = bw.patch(id, patches[id]);
    }
  }
  return results;
};
```

#### 3.3.3 Usage in a component

```javascript
function makeCounter(opts) {
  var id = bw.uuid('ctr');
  var valId = bw.uuid('val');    // stable address for the mutable value

  return {
    t: 'div', a: { id: id },
    o: {
      state: { count: opts.start || 0 },
      render: function(el) {
        var s = el._bw_state;
        bw.DOM(el, {
          t: 'div', a: { class: 'ctr-card' },
          c: [
            { t: 'h4', c: opts.label },
            { t: 'div', a: { id: valId, class: 'ctr-value' }, c: String(s.count) },
            { t: 'div', a: { class: 'ctr-btns' }, c: [
              bw.makeButton({ text: '+',
                onclick: function() {
                  s.count++;
                  bw.patch(valId, String(s.count));  // targeted update
                }
              }),
              bw.makeButton({ text: 'Reset',
                onclick: function() {
                  s.count = opts.start || 0;
                  bw.update(el);                     // full re-render (reset may change structure)
                }
              })
            ]}
          ]
        });
      }
    }
  };
}
```

**Key insight**: The developer chooses the update strategy per action:
- **Value change** (count incremented): `bw.patch(valId, newValue)` — fast, no rebuild
- **Structural change** (reset, add item, remove item): `bw.update(el)` — full re-render

## 4. Two-Tier Update Model

| Change type | Method | What happens | When to use |
|---|---|---|---|
| Value (text, attribute) | `bw.patch(id, value)` | Updates one element by UUID | Counter display, status text, badge count |
| Attribute | `bw.patch(id, value, 'class')` | Updates one attribute by UUID | Toggle active class, change color |
| Structural | `bw.update(el)` | Calls `el._bw_render(el)` → full rebuild | Add/remove list items, change layout |
| External | `bw.update(el)` | Same, but from outside the component | Cross-component coordination |

### 4.1 Decision Flowchart

```
State changed
  │
  ├── Only a value changed (text, number, attribute)?
  │     └── bw.patch(uuid, newValue)
  │
  └── Structure changed (items added/removed, layout changed)?
        └── bw.update(el)   // or bw.DOM(el, newTaco)
```

### 4.2 Consistency Rule

After any update (patch or full), this invariant must hold:

```
el._bw_state  ===  what the DOM displays
```

This is testable: read `el._bw_state.count`, read `document.getElementById(valId).textContent`, assert they match. The state-debug page already tests this.

## 5. Cross-Component Coordination

### 5.1 Current Problem

Local buttons call `renderGlobal()` directly — the counter card knows about the global panel:

```javascript
onclick: function() {
  s.count++;
  render(el);
  renderGlobal();  // tight coupling
}
```

### 5.2 Event-Based Solution: `bw.on()` / `bw.emit()`

Lightweight pub/sub scoped to DOM elements:

```javascript
// On the component element — fire event after state change
bw.emit(el, 'statechange', { count: s.count });

// On the global panel — listen for changes on any card
allCards.forEach(function(c) {
  bw.on(c.getElement(), 'statechange', function() {
    renderGlobal();
  });
});
```

**Implementation** — thin wrapper over DOM CustomEvent:

```javascript
bw.emit = function(target, eventName, detail) {
  var el = typeof target === 'string' ? document.querySelector(target) : target;
  if (el) {
    el.dispatchEvent(new CustomEvent('bw:' + eventName, {
      bubbles: true,
      detail: detail || {}
    }));
  }
};

bw.on = function(target, eventName, handler) {
  var el = typeof target === 'string' ? document.querySelector(target) : target;
  if (el) {
    el.addEventListener('bw:' + eventName, function(e) {
      handler(e.detail, e);
    });
  }
  return el;
};
```

**Why `bubbles: true`**: A parent can listen for statechange on any descendant. The global panel can listen on a shared container instead of wiring to each card individually:

```javascript
bw.on('#cards-mount', 'statechange', function(detail) {
  renderGlobal();
});
```

**Why `bw:` prefix**: Avoids collision with native DOM events. `bw:statechange` won't interfere with any browser event.

### 5.3 Automatic `statechange` Emission

When `bw.update(el)` is called, it could automatically emit `bw:statechange`:

```javascript
bw.update = function(target) {
  var el = typeof target === 'string' ? document.querySelector(target) : target;
  if (el && el._bw_render) {
    el._bw_render(el, el._bw_state || {});
    bw.emit(el, 'statechange', el._bw_state);
  }
  return el;
};
```

**Open question**: Should `bw.patch()` also emit `statechange`? Probably not automatically — `bw.patch` is a low-level DOM operation that doesn't know about component state. The developer should emit manually if needed, or use `bw.update()` which does both.

### 5.4 Topic-Based Pub/Sub: `bw.pub()` / `bw.sub()` / `bw.unsub()`

**Status**: Implemented (March 2026)

While `emit/on` is ideal for DOM-scoped events (parent-child coordination, bubbling), some events are **logical** — composite conditions, cross-cutting concerns, or application-wide notifications that don't map to a DOM subtree. For these, bitwrench provides a topic-based pub/sub layer.

**Two systems coexist:**

| | `emit/on` | `pub/sub` |
|---|---|---|
| Scope | DOM subtree (bubbles) | Application-wide (topic) |
| Trigger | 1:1 with DOM action | Composite condition / logic |
| Cleanup | Automatic (DOM GC) | `unsub()` or element-tied |
| Coupling | Spatial (ancestor) | None (any component) |
| Error | Native DOM throws | Try/catch, log, continue |

#### `bw.pub(topic, detail)` — Publish to a topic

```javascript
bw.pub('gameover', { winner: 'X', score: 3 });
```

- Iterates subscribers in registration order
- Try/catch per subscriber: `console.warn()` on error, continues to remaining
- Uses `subs.slice()` snapshot so unsubs during iteration are safe
- Returns count of successfully called subscribers
- Topic names are plain strings (no `bw:` prefix)

#### `bw.sub(topic, handler, el?)` — Subscribe to a topic

```javascript
// Basic subscription — returns unsub function
var unsub = bw.sub('gameover', function(detail) {
  console.log(detail.winner + ' won!');
});

// Element-tied — automatically unsubscribes when element is cleaned up
bw.sub('notification', function(detail) {
  messages.push(detail.text);
}, panelElement);
```

- Returns an `unsub()` function (removes by unique internal ID)
- Optional 3rd arg `el`: ties subscription to element lifecycle
  - Stores unsub in `el._bw_subs` array
  - Ensures element has `data-bw-id` (so `bw.cleanup()` finds it)
  - When `bw.cleanup(el)` is called, all tied subscriptions are removed
- `bw.DOM()` preserves `_bw_subs` on the mount element across re-renders

#### `bw.unsub(topic, handler)` — Unsubscribe by reference

```javascript
var handler = function(d) { /* ... */ };
bw.sub('topic', handler);
bw.unsub('topic', handler);  // removes ALL instances of handler on that topic
```

- Removes by handler reference equality (`===`)
- Returns count of removed subscriptions
- Cleans empty topic arrays from `_topics`

#### Internal Registry

```javascript
bw._topics = {};        // plain object (IE11 compat), topic → [{handler, id}]
bw._subIdCounter = 0;   // monotonic ID for subscriptions
```

#### When to Use Which

- **emit/on**: Child component signals parent. `bw.update()` emits `bw:statechange`. DOM cleanup is automatic.
- **pub/sub**: Game signals scoreboard. File-loaded AND terms-agreed → ready. Audit logging. Any cross-cutting concern.

#### Example: Both Systems Coexisting

```javascript
// Card button handler uses both systems:
onclick: function() {
  s.count++;
  bw.update(el);                          // emits bw:statechange (DOM)
  bw.pub('counter.audit', {               // publishes to topic
    card: name, action: '+1', value: s.count
  });
}

// Global panel listens via DOM events (parent-child):
bw.on('#cards-mount', 'statechange', function() {
  renderGlobal();
});

// Audit log subscribes via pub/sub (cross-cutting):
bw.sub('counter.audit', function(detail) {
  auditEntries.push(detail);
  renderAuditLog();
});
```

## 6. The List Problem

### 6.1 Challenge

Given a list of items:

```javascript
var items = [
  { id: bw.uuid('item'), text: 'Alpha' },
  { id: bw.uuid('item'), text: 'Beta' },
  { id: bw.uuid('item'), text: 'Gamma' }
];
```

Rendered as:

```html
<ul>
  <li id="bw_item_aaa">Alpha</li>
  <li id="bw_item_bbb">Beta</li>
  <li id="bw_item_ccc">Gamma</li>
</ul>
```

If Beta is removed, the DOM becomes:

```html
<ul>
  <li id="bw_item_aaa">Alpha</li>
  <li id="bw_item_ccc">Gamma</li>
</ul>
```

`bw_item_ccc` still addresses Gamma, regardless of its position changing from index 2 to index 1.

### 6.2 Item Operations

| Operation | Method | Addressing |
|---|---|---|
| Update item text | `bw.patch('bw_item_ccc', 'New text')` | UUID (stable) |
| Remove item | Modify state array, `bw.update(listEl)` | Full re-render |
| Add item | Modify state array, `bw.update(listEl)` | Full re-render |
| Reorder items | Modify state array, `bw.update(listEl)` | Full re-render |
| Toggle item class | `bw.patch('bw_item_ccc', 'done', 'class')` | UUID (stable) |

**Key rule**: Structural changes (add/remove/reorder) use `bw.update()`. Value changes within existing items use `bw.patch()`.

### 6.3 Where UUIDs Live

UUIDs are generated when **data** is created, not when DOM elements are rendered:

```javascript
function addTodo(el, text) {
  var s = el._bw_state;
  s.todos.push({
    id: bw.uuid('item'),    // UUID created here, in the data
    text: text,
    done: false
  });
  bw.update(el);             // re-render uses item.id as the <li> id
}
```

The render function maps data UUIDs to DOM element IDs:

```javascript
s.todos.map(function(item) {
  return {
    t: 'li',
    a: { id: item.id, class: item.done ? 'done' : '' },
    c: item.text
  };
})
```

This means:
- UUIDs travel with the data, not the DOM
- Server-generated JSON can include UUIDs — client renders them as element IDs
- Cloning state (e.g., undo/redo) preserves addressing

## 7. TACO → HTML vs. TACO → DOM

### 7.1 Two Code Paths

Bitwrench has two ways to convert a TACO to output:

| | `bw.html(taco)` | `bw.createDOM(taco)` |
|---|---|---|
| Output | HTML string | DOM nodes |
| Environment | Browser + Node.js | Browser only |
| Use case | SSR, innerHTML, email templates | Client-side UI, interactive components |
| State | None (stateless string) | `el._bw_state` on live elements |
| Events | Inline `onclick="..."` strings (limited) | `addEventListener` (full) |

### 7.2 How Addressing Works in Each Path

**TACO → HTML** (`bw.html`):
```javascript
bw.html({ t: 'span', a: { id: 'bw_val_abc' }, c: '42' })
// → '<span id="bw_val_abc">42</span>'
```

After insertion into the DOM (via innerHTML), `document.getElementById('bw_val_abc')` works. `bw.patch('bw_val_abc', '43')` works. The HTML is inspectable via view-source.

**TACO → DOM** (`bw.createDOM`):
```javascript
var el = bw.createDOM({ t: 'span', a: { id: 'bw_val_abc' }, c: '42' });
// el is a live DOM node with id="bw_val_abc"
```

Same addressing, same `bw.patch()`. No difference to the consumer.

### 7.3 The "Compiled Code" Question

The user raised: "we need to show all the compiled code (fine) which is different than react where this is never generated."

With React, the "compiled" output is vDOM nodes — opaque JavaScript objects that are never visible in the source. The only way to see what React rendered is to inspect the live DOM.

With bitwrench:
- **TACO → HTML**: The compiled output is a literal HTML string. Inspectable, copyable, debuggable. `bw.html(taco)` returns something you can paste into an HTML file.
- **TACO → DOM**: The compiled output is real DOM nodes with real IDs. DevTools shows them. `bw_val_abc` is right there in the Elements panel.

**This is a feature.** Bitwrench's output is always transparent. There's no hidden reconciliation layer. What you see in the DOM IS the state of the application.

UUIDs make this even more transparent: every mutable value has a visible, stable ID. You can test it: `document.getElementById('bw_val_abc').textContent === '42'`.

## 8. Edge Cases & Open Questions

### 8.1 Nested Components

If component A contains component B, and A does `bw.update(elA)`:
- A's entire subtree is rebuilt (including B)
- B's `_bw_state` is on B's element, which is destroyed
- B's state is lost

**Solution**: Components own their own mount points. A's render should not rebuild B's mount:

```javascript
// A's render:
bw.DOM(elA, {
  t: 'div', c: [
    { t: 'h2', c: 'Parent' },
    { t: 'div', a: { id: 'b-mount' } }   // B mounts here, A doesn't touch it
  ]
});
// B mounts separately:
bw.DOM('#b-mount', makeComponentB());
```

If A needs to re-render, it should use `bw.patch` for its own values and leave B's mount point alone. If A truly needs to rebuild its structure (including B's mount), it must re-mount B afterwards.

**Open question**: Should `bw.update(el)` skip children that have their own `_bw_state`? This would be a form of component boundary — "don't re-render below this point." Worth prototyping.

### 8.2 UUID Collision

`bw.uuid()` uses `crypto.randomUUID()` (128-bit). Collision probability is negligible for any practical page. Not a real concern.

### 8.3 Server-Generated UUIDs

If a server generates TACO JSON with UUIDs, the client must use those UUIDs (not generate new ones). `bw.createDOM` already uses `id` from the `a` (attributes) object if provided. No change needed.

### 8.4 Stale Patches

If `bw.patch('bw_val_abc', '43')` is called but the element was already destroyed (e.g., parent re-rendered), `getElementById` returns null. `bw.patch` returns null. No error, no side effect. This is fine — the value will be correct on the next full render.

### 8.5 `bw.patch` vs. `el._bw_state` Consistency

`bw.patch` updates the DOM but does NOT update `el._bw_state`. The developer must update state first, then patch:

```javascript
s.count++;                           // update state
bw.patch(valId, String(s.count));    // update DOM to match
```

This matches the mental model: state is the source of truth, DOM reflects state. `bw.patch` is a DOM operation, not a state operation.

**Alternative considered**: `bw.patch` could accept a state path and update both. Rejected — it adds complexity and couples `bw.patch` to the state model. Keep them separate.

## 9. Implementation Order

1. **`o.render`** — store as `el._bw_render`, preserve in `bw.DOM()`, call on mount
2. **`bw.update(el)`** — call `el._bw_render(el)`, emit `bw:statechange`
3. **`bw.patch(id, content)`** — targeted DOM update by UUID
4. **`bw.patchAll(patches)`** — batch version
5. **`bw.emit()` / `bw.on()`** — lightweight pub/sub via CustomEvent
6. **Update state-debug page** — demonstrate all three in the counter/global example
7. **Update 05-advanced-features page** — use new patterns as best practices
8. **Tests** — extend `test/state-debug.spec.js` and `test/mounted-pattern.spec.js`

## 10. Summary

| Concept | Mechanism | Key Property |
|---|---|---|
| Component identity | `bw.uuid('ctr')` → element ID | CSS-selectable, stable |
| Mutable addressing | `bw.uuid('val')` → element ID | Survives reorder, inspectable |
| State storage | `el._bw_state` | Tied to element lifecycle |
| Render function | `el._bw_render` (from `o.render`) | Preserved across re-renders |
| Value update | `bw.patch(uuid, value)` | Targeted, no rebuild |
| Structural update | `bw.update(el)` | Full re-render via `_bw_render` |
| Cross-component | `bw.emit()` / `bw.on()` | Bubbling CustomEvents, `bw:` prefix |
| Consistency invariant | `el._bw_state` === DOM display | Testable after every operation |
