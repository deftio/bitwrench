# v2.0.18 Styles Cleanup — Design Doc

**Status:** Draft — discussion
**Date:** 2026-03-15
**Goal:** A dead-simple mental model for CSS in bitwrench.

---

## The Problem

The current CSS system has three competing layers with no clear hierarchy:

1. `bw.loadDefaultStyles()` — structural + cosmetic via `generateTheme('', ...)`
2. `bw.generateTheme('name', config)` — scoped CSS + class on `<body>` + side effects
3. Hand-written `bw.css()` / `bw.injectCSS()` — page-specific overrides

These interact in unpredictable ways. The nav-link bug (v2.0.18) was caused by
`generateTheme('dashboard-theme', ...)` adding a class to `<body>`, making
`.dashboard-theme a { color: ... }` (0,1,1) beat `.bw_site_nav_link` (0,1,0).

### What the audit found

**Bare element selectors emitted by the style system:**

| Source | Bare selectors |
|--------|---------------|
| Structural (reset) | `*`, `html`, `body`, `main`, `hr`, `h1`–`h6`, `p`, `small`, `a` |
| Themed (cosmetic) | `body` (color/bg), `a` (link color), `a:hover` |
| Utilities | `.fs-*`, `.fw-*`, `.list-unstyled`, `.pe-none`, `.opacity-*` (no `bw_` prefix) |
| Grid | `[class*="col-"]` attribute selector (matches anything with "col-" in class) |

These cause specificity conflicts when themes are scoped under a class on `<body>`.

**Side effects in `generateTheme()`:**
- Injects `<style>` elements (2: primary + alternate)
- Adds theme class to `<body>`
- Sets `bw._activeTheme`, `bw._activeThemeMode`, `bw._activeThemeStyleIds`, `bw._activeThemeClass`
- Mutates `bw.u` color entries when name is `''`

One function doing generation, injection, DOM mutation, and internal state management.

### What QuikChat gets right

QuikChat's CSS model is trivially simple:
- All selectors namespaced: `.quikchat-messages-area`, `.quikchat-input-send-btn`
- Theme switching = class swap on the widget root element (not `<body>`)
- Zero bare element selectors
- JS never generates CSS — the CSS is static and external
- User can write custom theme CSS by targeting `.my-theme .quikchat-*` selectors

Bitwrench can't be identical (we generate CSS from JS — that's a feature), but we can
adopt the same separation of concerns and scoping discipline.

## Structural vs Styled CSS

Every BCCL component's CSS has two distinct layers:

**Structural CSS** — makes the component *work* as a component:
- `display`, `flex-direction`, `align-items` (layout)
- `overflow: hidden` (card clips content)
- `border-collapse: collapse` (table cells join)
- `position: fixed/absolute` (modal, tooltip positioning)
- `cursor: pointer` (interactive elements)
- `list-style: none` (nav items)
- `white-space: nowrap`, `text-overflow: ellipsis` (truncation)
- `box-sizing: border-box` (everything)
- Minimal spacing where needed for usability (a button needs *some* padding
  to be clickable, a card needs content not flush to edges)

**Styled CSS** — makes the component *beautiful*:
- Colors, backgrounds, gradients
- `border-radius`, `box-shadow`
- `border-color`, `border-width`
- Specific spacing amounts (design tokens from the spacing scale)
- `font-size`, `font-weight`, `font-family`
- `transition`, `opacity`, hover/focus effects
- Line thickness, text decoration

### The rule

**BCCL components have built-in structural CSS. They have NO styled CSS.**

Styled CSS comes entirely from the theme/styles system. If you load no styles,
components still work — correct layout, clickable buttons, collapsible tables —
but they look plain (browser defaults). Load a theme and they become gorgeous.

### Gray area: spacing

Some padding IS structural — a zero-padding button is unusable. Judgment applies:
- Structural spacing uses relative units (`0.4em`, `0.8em`) so it scales with context
- Themed spacing uses the spacing scale tokens (`compact`/`normal`/`spacious`)
  and overrides structural defaults with specific `rem` values
- If no theme is loaded, structural spacing keeps things usable

### Impact on `bitwrench-styles.js`

Currently `getStructuralStyles()` mixes structural and cosmetic rules. Under this
model, it splits cleanly:

| Layer | Contents | When loaded | Varies with theme? | Est size |
|-------|----------|-------------|-------------------|----------|
| **Structural base** | display, flex, overflow, cursor, box-sizing, minimal spacing | Inline on templates (always) + class-based via `makeStyles()` | No | ~1KB |
| **Themed cosmetic** | colors, radius, shadows, borders, spacing tokens, fonts | Via `makeStyles()` + `applyStyles()` | Yes | ~8-12KB |
| **CSS reset** | `*`, `html`, `body` box-sizing/font-size | Once per page, optional | No | ~0.5KB |

The structural base is stable across all themes — ocean, sunset, custom, or none.

## BCCL Components as Inspectable TACO Templates

### No magic factories

The current `make*()` functions are opaque — you call `makeCard({...})` and get a
TACO back, but there's no way to see the base template, understand what classes are
emitted, or learn how to build your own.

Under the clean model, every BCCL component is a plain TACO object stored in a
registry. `make*()` is just sugar that pulls the template and fills in your attributes.

```javascript
// The registry — just a map of TACO templates.
// Empty fields omitted (saves space). Structural CSS inline where fixed.
bw.components = {
  card: {
    t: 'div',
    a: { class: 'bw_card', style: { display: 'flex', flexDirection: 'column', overflow: 'hidden' } },
    c: [
      { t: 'div', a: { class: 'bw_card_header' }, o: { slot: 'title' } },
      { t: 'div', a: { class: 'bw_card_body' },   o: { slot: 'content' } }
    ]
  },
  button: {
    t: 'button',
    a: { class: 'bw_btn', type: 'button', style: { cursor: 'pointer' } },
    o: { slot: 'label' }    // slot defaults to field 'c'
  },
  // ... all 27 components
};

// Slots: o.slot names a fill point. make*() maps props to slots.
// Default field is 'c' (content), but can target other fields:
//   o: { slot: 'icon', field: 'a.class' }  — fill into attribute instead
// This makes make*() a generic hydrator — one function for all components.

// makeCard() is thin sugar over the registry
bw.makeCard = function(props) {
  var base = bw.deepClone(bw.components.card);
  // generic: walk template, find o.slot entries, fill from props
  // e.g. props.title → slot 'title' → card_header.c
  //      props.content → slot 'content' → card_body.c
  return base;
};
```

### Why this matters

1. **Inspectable.** A developer can do `console.log(bw.components.card)` and see
   exactly what a card IS — it's a `{t,a,c,o}`. No magic, no hidden behavior.

2. **Forkable.** Want a card with different structure? Clone the template, modify it,
   register it under your own name or just use it directly.

3. **Learnable.** The pattern for building your own components is obvious: make a
   TACO, give it a class, write CSS for that class. `make*()` is a convenience,
   not a requirement.

4. **CSS-independent.** The templates carry structural classes. Whether those classes
   have styled CSS attached depends on whether you loaded a theme. The component
   works either way.

5. **Portable.** A developer can export their component registry as JSON, save it,
   share it, or load a different set of templates at runtime.

### Getting a blank component

```javascript
// Get the bare template — just structure, no content filled in
var blankCard = bw.makeComponent('card');
// Returns: { t: 'div', a: { class: 'bw_card' }, c: [...] }

// Inspect it, modify it, learn from it
blankCard.a.class += ' my_custom_class';
blankCard.c[0].c = 'My Header';

// Use it directly — no make*() needed
bw.DOM('#app', blankCard);
```

### Proprietary theme makers

Because BCCL templates are just TACO objects with `bw_*` classes, a developer can:
1. Use bitwrench's `makeStyles()` to generate CSS for those classes (easy path)
2. Write their own CSS targeting `bw_*` classes (full control)
3. Use `prefix: 'myapp_'` to generate a parallel set of classes (isolation)
4. Mix and match: use bitwrench's card structure but their own button styling
5. Build their own `makeStyles()`-equivalent that generates CSS from their own
   design tokens — bitwrench shows the *pattern* (JS → CSS), not the only way

The point: bitwrench demonstrates JS-based theme and style scoping as *a* way —
not *the* way — to make maintainable web content. Take what you want, leave the rest.

## The Model

Two operations. That's it.

### Step 1: MAKE styles (generation)

The user creates CSS somehow. Three paths:

```
Path A: Use bitwrench theme generator
  var styles = bw.makeStyles({ primary: '#4f46e5', secondary: '#d97706' });

Path B: Use bitwrench helpers to build custom CSS
  var css = bw.css({ '.my-class': { color: 'red' } });

Path C: Write your own CSS however you want
  (Tailwind, raw CSS file, inline styles — bitwrench doesn't care)
```

**Naming rationale:** `makeStyles` not `getStyles`. It *creates* CSS from a config,
same as `makeCard` creates a TACO from props. `get` implies retrieving something
that already exists.

#### `bw.makeStyles()` — params and return value

```javascript
var styles = bw.makeStyles({
  primary:    '#4f46e5',         // optional — default '#006666' (bitwrench teal)
  secondary:  '#d97706',        // optional — default '#cc6633'
  tertiary:   '#10b981',        // optional — defaults to primary
  background: '#fafafe',        // optional — page background tint (hue-matched to primary)
  surface:    '#f5f3ff',        // optional — card/panel backgrounds
  spacing:    'normal',         // optional — 'compact' | 'normal' | 'spacious'
  radius:     'md',             // optional — 'none' | 'sm' | 'md' | 'lg' | 'pill'
  globals:    false,            // optional — emit bare element selectors (a, h1, body)?
  prefix:     'bw_'             // optional — class prefix (default 'bw_'). Reserved, not yet implemented.
});
```

Returns a plain object — no side effects, no DOM, no state:

```javascript
{
  css: '.bw_btn_primary { background: #4f46e5; color: #fff; }\n' +
       '.bw_btn_primary:hover { background: #3730a3; }\n' +
       '.bw_card { border: 1px solid #c7c3f9; border-radius: 0.375rem; }\n' +
       '.bw_card_header { background: #eef2ff; border-bottom: 1px solid #c7c3f9; padding: 0.75rem 1rem; }\n' +
       '/* ... ~200 more rules for all BCCL components ... */',

  alternateCss: '.bw_btn_primary { background: #818cf8; color: #1e1b4b; }\n' +
                '.bw_card { border-color: #312e81; background: #1e1b4b; }\n' +
                '/* ... alternate palette rules (no .bw_theme_alt prefix — raw rules) ...\n' +
                '   applyStyles() wraps these with .bw_theme_alt (unscoped)\n' +
                '   or #scope.bw_theme_alt (scoped) at injection time */',

  palette: {
    primary:   { base: '#4f46e5', hover: '#3730a3', active: '#312e81',
                 light: '#eef2ff', darkText: '#1e1b4b', border: '#c7c3f9',
                 focus: 'rgba(79,70,229,0.25)', textOn: '#fff' },
    secondary: { base: '#d97706', hover: '#b45309', active: '#92400e',
                 light: '#fef3c7', darkText: '#78350f', border: '#fcd34d',
                 focus: 'rgba(217,119,6,0.25)', textOn: '#fff' },
    success:   { base: '#16a34a', /* ... 8 keys ... */ },
    danger:    { base: '#dc2626', /* ... */ },
    warning:   { base: '#eab308', /* ... */ },
    info:      { base: '#0891b2', /* ... */ },
    light:     { base: '#f8fafc', /* ... */ },
    dark:      { base: '#1e1b4b', /* ... */ },
    background: '#fafafe',
    surface:    '#f5f3ff'
  },

  alternatePalette: { /* same structure, luminance-inverted */ },

  isLightPrimary: true
}
```

You can call `makeStyles()` 100 times — nothing happens to the page. It's data.

### Step 2: APPLY styles (injection)

The user puts CSS into the DOM with explicit scoping:

```javascript
bw.applyStyles(styles);                    // global (primary as-is + alternate wrapped with .bw_theme_alt)
bw.applyStyles(styles, '#my-dashboard');   // scoped to container
bw.applyStyles(styles, '.preview-area');   // scoped to preview
```

#### What `applyStyles()` actually does, precisely:

1. Creates a `<style>` element in `<head>` (via `injectCSS()` internally)
2. Gives it a trackable id: `bw_style_<scope>` (e.g. `bw_style_global`,
   `bw_style_my_dashboard`). Findable via `document.getElementById(...)`.
3. If a scope selector is provided (e.g. `'#my-dashboard'`), wraps every
   primary CSS rule under that selector:
   `.bw_card { ... }` → `#my-dashboard .bw_card { ... }`
4. Wraps alternate CSS with `.bw_theme_alt`:
   - **Unscoped:** `.bw_card { ... }` → `.bw_theme_alt .bw_card { ... }`
   - **Scoped:** `.bw_card { ... }` → `#my-dashboard.bw_theme_alt .bw_card { ... }`
5. Concatenates primary + wrapped alternate into one CSS string
6. Sets the `<style>` element's `textContent` to the combined CSS
7. If a `<style>` with that id already exists, **replaces** its content (idempotent)
8. Returns the `<style>` element

No body classes. No internal state mutations.

To remove styles later:

```javascript
bw.clearStyles('#my-dashboard');
// Finds <style id="bw_style_my_dashboard">, removes it.
```

### Convenience: MAKE + APPLY in one call

For the common case where you want to generate and apply in one step:

```javascript
bw.loadStyles();                                          // defaults, global
bw.loadStyles({ primary: '#4f46e5' });                    // custom colors, global
bw.loadStyles({ primary: '#4f46e5' }, '#my-dashboard');   // custom, scoped
```

`bw.loadStyles(config, scope)` = `bw.applyStyles(bw.makeStyles(config), scope)`.
Sugar, not magic. Literally two function calls composed into one — no third
system, no additional concept. It replaces `loadDefaultStyles()` as the
zero-config entry point. If you need the palette object (for custom CSS using
palette colors), use `makeStyles()` + `applyStyles()` separately.

### No implicit behavior

Nothing happens behind your back. Specifically:

- `bw.DOM()` does NOT auto-load styles. If you haven't loaded styles, components
  render with structural CSS only (functional but unstyled). This is correct
  behavior, not a bug.
- `make*()` does NOT inject CSS. It returns a TACO. Period.
- `makeStyles()` does NOT touch the DOM. It returns data.
- `applyStyles()` does exactly one thing: injects a `<style>`. No body classes,
  no internal state mutations.

The current `loadDefaultStyles()` violates all of this — it calls
`generateTheme('', ...)` which injects CSS, mutates `bw.u`, sets
`bw._activeTheme`, and adds a class to `<body>`. That pattern goes away entirely.

### Alternate / dark palette

Every theme has two sides: primary and alternate (luminance-inverted).
Both are created by `makeStyles()` and returned in the result.

```javascript
var styles = bw.makeStyles({ primary: '#4f46e5', secondary: '#d97706' });

// Apply styles (injects BOTH primary + alternate CSS in one <style>)
bw.applyStyles(styles);

// Toggle between primary ↔ alternate
bw.toggleStyles();                  // global — flips class on <body>
bw.toggleStyles('#my-dashboard');   // scoped — flips class on that element
```

`applyStyles()` second arg is always a scope selector string (or omitted for
global). There is no `{ mode: 'alternate' }` options object — toggling is done
by `toggleStyles()`, not by calling `applyStyles()` again.

**How alternate works internally:** `applyStyles()` injects both `styles.css`
and `styles.alternateCss` into the same `<style>` element. The alternate rules
are scoped under `.bw_theme_alt`:

```css
/* Primary (always active) */
.bw_card { background: #fff; border: 1px solid #c7c3f9; }

/* Alternate (active when .bw_theme_alt is present) */
.bw_theme_alt .bw_card { background: #1e1b4b; border: 1px solid #312e81; }
```

**Where `.bw_theme_alt` goes:**
- **Global** (no scope): `toggleStyles()` adds/removes `.bw_theme_alt` on `<body>`.
- **Scoped**: `toggleStyles('#my-dashboard')` adds/removes `.bw_theme_alt` on
  the `#my-dashboard` element. The alternate CSS for scoped case uses a compound
  selector: `#my-dashboard.bw_theme_alt .bw_card` (class ON the scoping element,
  not as a descendant).

The key difference: unscoped alternate CSS uses `.bw_theme_alt .bw_card`
(descendant of `<body>`). Scoped alternate CSS uses `#scope.bw_theme_alt .bw_card`
(compound on the scope element).

**Implementation detail:** `makeStyles()` returns `alternateCss` WITHOUT the
`.bw_theme_alt` prefix — just raw alternate rules (`.bw_card { background: #1e1b4b; }`).
`applyStyles()` wraps them appropriately:
- **Unscoped:** prepend `.bw_theme_alt ` to each selector
  → `.bw_theme_alt .bw_card { background: #1e1b4b; }`
- **Scoped:** prepend `#scope.bw_theme_alt ` to each selector
  → `#my-dashboard.bw_theme_alt .bw_card { background: #1e1b4b; }`

This keeps `makeStyles()` pure (no scoping knowledge) and puts all wrapping
logic in one place (`applyStyles()`).

## The Scoping Rule

**Default: ALL generated CSS targets only `bw_*` class selectors.**

```css
/* YES — generated by default */
.bw_btn_primary { background: #4f46e5; color: #fff; }
.bw_card { border: 1px solid #e5e7eb; }
.bw_table th { background: #f8f9fa; }

/* NO — never generated by default */
a { color: #4f46e5; }
h1 { color: #1a1a1a; }
body { background: #fff; }
```

This means:
- bitwrench components (`make*()`) always look right
- user's own HTML is never touched
- multiple themes coexist without specificity wars
- the nav bug can never happen

### Opt-in global element styling

If a user WANTS their `<h1>`s and `<a>`s themed:

```javascript
var styles = bw.makeStyles({
  primary: '#4f46e5',
  globals: true    // also emit a, h1-h6, body, etc.
});
```

The global rules are included in the same `styles.css` output — no separate
function needed. If someone wants just the global rules alone, they can write
`bw.css({ 'a': { color: styles.palette.primary.base } })` by hand.

### CSS Reset

The structural reset (`*`, `html`, `body`, `box-sizing`) is a separate concern.
It's needed once per page, not per theme.

```javascript
bw.loadReset();
// Injects box-sizing, html font-size, etc. — once
// Creates: <style id="bw_style_reset"> in <head>
// Detectable: document.getElementById('bw_style_reset')
// Removable: bw.clearStyles('reset')
```

The reset is small (~20 rules), non-destructive, and idempotent.
If `<style id="bw_style_reset">` already exists, `loadReset()` returns the
existing element without modifying it (idempotent).

## Custom Class Prefix

The default prefix is `bw_`. A developer can use their own:

```javascript
var styles = bw.makeStyles({
  primary: '#4f46e5',
  prefix: 'myapp_'    // all classes become myapp_btn_primary, myapp_card, etc.
});
```

This is critical for:
- Embedding bitwrench widgets in a host page without CSS conflicts
- Building white-label products where "bw_" branding is undesirable
- Co-existing with other bitwrench instances using different themes

The `make*()` functions would need to know the active prefix to emit the right
classes. This requires parameterizing every class string in `bitwrench-styles.js`
and `bitwrench-bccl.js` — mechanically invasive.

**Decision: punt to v2.1.0.** The `prefix` field is reserved in the `makeStyles()`
config so the API shape is ready, but the plumbing isn't wired yet. For v2.0.18,
prefix is always `bw_`. Developers who need a custom prefix today can do string
replacement on the CSS output and adjust their TACO class strings — bitwrench
shows the pattern, they take over from there.

## Migration Path

### What stays
- `bw.css()` — pure function, generates CSS from JS objects. No changes needed.
- `bw.injectCSS()` — low-level DOM injection. Stays as internal plumbing.
- `bw.s()` — inline style merge. 6 lines, general-purpose `Object.assign` wrapper.
  Stays in core but demoted from docs spotlight.
- `bw.responsive()` — pure function. No changes.

### What changes
| Current | New | Notes |
|---------|-----|-------|
| `bw.loadDefaultStyles()` | `bw.loadStyles(config?, scope?)` | Convenience: make + apply. |
| `bw.generateTheme(name, config)` | `bw.makeStyles(config)` | Pure function. No injection, no body class, no side effects. |
| `bw.applyTheme(mode)` | `bw.toggleStyles(scope?)` | Toggle alternate on scoping element (was: toggle on `<html>`). |
| `bw.toggleTheme()` | `bw.toggleStyles(scope?)` | Toggle alternate on scoping element, not `<html>`. |
| `bw.clearTheme()` | `bw.clearStyles(scope?)` | Remove injected styles for a scope. |
| `bw.setTheme()` | removed | Was already deprecated. |

### No deprecation wrappers — hard rename

No one is using v2.x in production. Old function names are removed, not deprecated.
Clean break.

## Impact on Existing Code

### BCCL components (`make*()`)
- Already emit `bw_*` classes — no change to output
- Extract base TACO templates into `bw.components` registry (inspectable map)
- `make*()` becomes thin sugar: clone template, fill in props, return TACO
- Add `bw.makeComponent(name)` to return a blank clone of any template
- Support custom prefix (passed through from active theme or explicit option)

### pages/ (docs site)
- `shared-theme.js` shrinks dramatically — site chrome layout only, no component styling
- Pages drop most `bw.injectCSS(bw.css({...}))` blocks
- One `bw.loadStyles()` call at the top of each page
- Nav CSS already fixed (specificity bump in this sprint)

### examples/
- `bw.loadDefaultStyles()` calls become `bw.loadStyles()`
- Examples that use `generateTheme()` switch to `bw.makeStyles()` + `bw.applyStyles()`
- Showcase example already works (zero custom CSS) — validates the model

### bitwrench-styles.js
- `getStructuralStyles()` — remove bare element selectors from default emission
  - Keep them available via `globals: true` option
  - CSS reset (`*`, `html`, `body` box-sizing) moves to separate `getResetStyles()`
- `generateThemedCSS()` — stop emitting `a`, `a:hover`, `body` as bare selectors
- `generateUtilityRules()` — add `bw_` prefixed versions of standalone Bootstrap-compat selectors (`.fs-*`, `.fw-*`, etc.) or drop them
- `generateAlternateCSS()` — scope under `.bw_theme_alt` within the scoping element, not on `<html>`

### bitwrench.js (public API)
- `generateTheme()` — becomes `makeStyles()`, pure function
- `applyTheme()` — becomes `applyStyles()`, explicit scoping
- `loadDefaultStyles()` — becomes `loadStyles()`, thin wrapper
- Remove body class manipulation from theme injection
- Remove `bw._activeThemeClass` body-class pattern entirely

## All CSS/Style Functions — Complete Reference

Every function related to CSS, styles, or themes. Grouped by role.
Pure functions have zero side effects. DOM functions list exactly what they touch.

### Pure functions (no side effects, no DOM)

| Function | Takes | Returns | What it does |
|----------|-------|---------|-------------|
| `bw.makeStyles(config?)` | `{ primary?, secondary?, tertiary?, background?, surface?, spacing?, radius?, globals?, prefix? }` | `{ css, alternateCss, palette, alternatePalette, isLightPrimary }` | **Theme generator.** Takes seed colors + options, returns complete CSS + palette data. All params optional — defaults to bitwrench palette. Calls `derivePalette()` → `generateThemedCSS()` → `bw.css()` internally. Can be called 100 times with no page effect. |
| `bw.css(rules, options?)` | JS object `{ '.sel': { prop: val } }` or array of same | CSS string | **CSS serializer.** Converts JS objects to CSS text. Supports `@media`, `@keyframes` nesting. No knowledge of themes or palettes — just object → string. |
| `bw.s(...styles)` | style objects (variadic) | merged style object | Merges inline style objects. `bw.s({ color: 'red' }, { padding: '1rem' })` → `{ color: 'red', padding: '1rem' }`. For use in TACO `a: { style: bw.s(...) }`. |
| `bw.responsive(breakpoints)` | `{ sm: rules, md: rules, lg: rules }` | CSS rules object with `@media` wrappers | Pure helper for responsive CSS. Returns an object suitable for passing to `bw.css()`. |
| `bw.derivePalette(config)` | `{ primary, secondary, tertiary, success?, ... }` | palette object (9 shade sets + bg + surface) | Color math. Derives 8 shades per seed color (base, hover, active, light, darkText, border, focus, textOn). |
| `bw.deriveShades(hex)` | hex color string | `{ base, hover, active, light, darkText, border, focus, textOn }` | Single-color shade derivation. Building block for `derivePalette()`. |
| `bw.makeComponent(name)` | component name (e.g. `'card'`, `'button'`) | TACO object (deep clone) | Returns a blank TACO template from the `bw.components` registry. No content filled in. Developer can inspect, modify, or use directly. |

### DOM functions (inject/remove `<style>` elements)

| Function | Takes | Returns | Side effects |
|----------|-------|---------|-------------|
| `bw.applyStyles(styles, scope?)` | styles object (from `makeStyles()`), optional scope selector string | `<style>` element | Creates `<style id="bw_style_{scope}">` in `<head>`. Wraps primary CSS under scope (if provided). Wraps alternate CSS under `.bw_theme_alt` (unscoped) or `scope.bw_theme_alt` (scoped, compound). Replaces existing `<style>` with same id (idempotent). |
| `bw.loadStyles(config?, scope?)` | same as `makeStyles()` config, optional scope | `<style>` element | Sugar: `applyStyles(makeStyles(config), scope)`. Two calls composed into one. |
| `bw.loadReset()` | — | `<style>` element | Injects `<style id="bw_style_reset">` with box-sizing, html font-size, etc. Idempotent — no-op if already present. ~20 rules, ~0.5KB. |
| `bw.toggleStyles(scope?)` | optional scope selector | `'primary'` or `'alternate'` | Adds/removes `.bw_theme_alt` class. No scope → on `<body>`. With scope → on the scoping element. Returns the new mode. |
| `bw.clearStyles(scope?)` | optional scope selector (default: `'global'`) | void | Finds `<style id="bw_style_{scope}">` via getElementById, removes it. No scope → removes `bw_style_global`. |
| `bw.injectCSS(css, options?)` | CSS string or rules object, `{ id?, append? }` | `<style>` element | Low-level plumbing. Creates/reuses `<style id="{id}">` in `<head>`. Used internally by `applyStyles()`. Developers CAN use directly for custom CSS injection. |

### Inline style utilities (no `<style>` elements, no DOM)

| Function | Takes | Returns | What it does |
|----------|-------|---------|-------------|
| `bw.s(...styles)` | style objects (variadic) | merged style object | Shallow merge via `Object.assign`. For use in TACO `a: { style: bw.s(base, override) }`. Same as pure functions table above. |

**`bw.u` (static dictionary) is removed from core.** Replaced by the `bw.utilCSS`
plugin — a rule-based parser in a separate file. See §`bw.utilCSS` Plugin below.

### What's gone (removed, not deprecated)

| Old function | Replacement | Why removed |
|-------------|-------------|------------|
| `bw.loadDefaultStyles()` | `bw.loadStyles()` | Opaque: called `generateTheme('', ...)` with 5+ side effects |
| `bw.generateTheme(name, config)` | `bw.makeStyles(config)` | Impure: injected CSS, added body class, mutated `bw.u`, set 4 internal vars |
| `bw.applyTheme(mode)` | `bw.toggleStyles(scope?)` | Manipulated `<html>` class globally |
| `bw.toggleTheme()` | `bw.toggleStyles(scope?)` | Same |
| `bw.clearTheme()` | `bw.clearStyles(scope?)` | Removed body class, reset internal vars — too many side effects |
| `bw.setTheme()` | removed | Was already deprecated |
| `bw.u` (static dict) | `bw.utilCSS` plugin | 54-entry static dictionary with hardcoded colors, mutated by `generateTheme()`. Replaced by rule-based parser in separate plugin. See §`bw.utilCSS`. |

### Example: full workflow

```javascript
// 1. Make styles (pure — no DOM touched)
var styles = bw.makeStyles({
  primary: '#4f46e5',
  secondary: '#d97706',
  spacing: 'compact',
  radius: 'md'
});

// 2. Apply to the whole page (injects one <style id="bw_style_global">)

// 3. Or apply scoped to a container

// Injects: <style id="bw_style_my_dashboard">
// Every rule wrapped: #my-dashboard .bw_card { ... }

// Adds class "bw_theme_alt" to #my-dashboard element

// Two themes coexist — no conflicts, different <style> elements

// Removes <style id="bw_style_alerts_panel"> from DOM

// { base: '#4f46e5', hover: '#3730a3', active: '#312e81',
//   light: '#eef2ff', darkText: '#1e1b4b', border: '#c7c3f9',
//   focus: 'rgba(79,70,229,0.25)', textOn: '#fff' }

// 8. Use palette colors in custom TACO
```

## Component Lifecycle

### What the lifecycle SHOULD be

First: the clean model. What stages exist, and what should happen at each one.

| # | Stage | What should happen | Who does it | Side effects |
|---|-------|--------------------|-------------|-------------|
| 1 | **Template** | A TACO skeleton exists: `{t, a, c, o}` with structural classes (`bw_card`, `bw_btn`). No content yet. | `bw.makeComponent('card')` returns a blank clone. Or developer writes TACO by hand. `make*(props)` is sugar that does stages 1+2 together. | None. Pure data. |
| 2 | **Props** | Content values merged in — title text, list items, variant class, event handlers. Still just a JS object. | `make*(props)` — destructure props, build children array, set class string. | None. Pure data. |
| 3 | **State** | If the component is reactive: initial state + render function attached. If static: skip this stage. | Developer sets `o: { state: {...}, render: fn }`. Or wraps in `bw.component(taco)` for reactive bindings. | None. Pure data. |
| 4 | **Styles** | CSS loaded so `bw_card`, `bw_btn_primary` etc. have visual rules. **Independent of stages 1-3 and 5-10.** Can happen before, after, or never. | `bw.makeStyles(config)` → `bw.applyStyles(styles, scope)`. Or developer writes own CSS. Or uses a `.css` file. | One `<style>` injected. Nothing else. |
| 5 | **DOM creation** | TACO → real DOM nodes. Elements created, attributes set, event listeners attached. Children recursed. | `bw.createDOM(taco)` — returns a detached DOM tree (not yet in document). | Nodes created. Event listeners bound. |
| 6 | **Registration** | Element gets indexed so it can be found later by id, UUID, or data-bw_id. | Automatic during creation for elements that have `id`, `data-bw_id`, or `bw_uuid_*` class. | Entries added to `bw._nodeMap`. |
| 7 | **Mount** | DOM tree appended to the live document. Element becomes visible and laid out. | `bw.DOM('#app', taco)` — clears target, appends new content. Or raw `parent.appendChild(el)`. | Previous children cleaned up (stage 10). New tree inserted. |
| 8 | **Hooks fire** | `o.mounted` or `o.render` called now that element is in the document and can be measured/queried. | Called by the mounting function, after appendChild. | Developer's hook code runs. May modify DOM, start timers, subscribe to events. |
| 9 | **Runtime** | Component updates in response to events, data, user interaction. | **Four patterns** (use cheapest that fits): (a) `bw.patch(id, val)` — one DOM write. (b) `bw.update(el)` — re-render subtree via `o.render`. (c) `bw.DOM('#x', newTaco)` — full teardown + rebuild. (d) `handle.set({k: v})` — reactive binding update, minimal DOM diff. | Depends on pattern chosen. |
| 10 | **Destruction** | Element removed from DOM. Resources freed: unmount hooks called, subs removed, cache entries cleared, timers stopped. | `bw.cleanup(el)` — walks descendants, fires unmount, deregisters. Called automatically by `bw.DOM()` before replacing. | Cleanup side effects only. |

**Key property: stages 1-3 are pure data. Stage 4 (styles) is independent of
everything else. Stages 5-8 are the DOM transition. Stage 9 is runtime. Stage 10
is cleanup. No stage should have hidden coupling to another.**

### What actually happens now

Map the current implementation onto these stages to see where reality diverges.

| # | Stage | Current implementation | Problems |
|---|-------|-----------------------|----------|
| 1 | **Template** | `makeCard(props)` returns TACO. No registry, no `makeComponent()`. | Opaque — can't inspect base template. Stages 1+2 always fused. |
| 2 | **Props** | Inside `make*()` — destructure, build, return. Pure. | Fine. |
| 3 | **State** | Developer writes `o: {state, render}` or uses `bw.component(taco)`. Two parallel systems. | `o.mounted` and `o.render` compete (warning emitted if both). `bw.component()` copies state out of TACO into its own `_state`, adding a layer. |
| 4 | **Styles** | `bw.loadDefaultStyles()` — calls `generateTheme('', config)` which: injects 2+ `<style>` elements, adds theme class to `<body>`, sets `bw._activeTheme` + 3 internal vars, mutates `bw.u` color entries. | **Entangled with component lifecycle.** Must be called before `bw.DOM()` for components to look right. Has 5+ side effects beyond injecting CSS. `generateTheme()` conflates generation + injection + DOM mutation + state mutation. |
| 5 | **DOM creation** | `bw.createDOM(taco)` — creates elements, sets attrs, event listeners. Also: registers in `bw._nodeMap` (stage 6), builds `_bw_refs` parent→child map, fires `o.mounted`/`o.render` via rAF (stage 8). | **Stages 5+6+8 collapsed into one call.** Developer can't create DOM without also registering and firing hooks. |
| 6 | **Registration** | Inside `createDOM()` — auto-registers by id, data-bw_id, bw_uuid_* class. 5-step lookup cascade in `bw._el()`. | Implicit — developer doesn't opt in, can't opt out. The 5-step cascade (`_nodeMap` cache → `getElementById` → `querySelector` → `data-bw_id` attr → `bw_uuid_*` class) is complex and order-dependent. |
| 7 | **Mount** | `bw.DOM(sel, taco)` — saves/restores target's `_bw_state`/`_bw_render`/`_bw_subs`/`data-bw_id`, calls `bw.cleanup()` on children, clears innerHTML, calls `createDOM()`, appends. For ComponentHandle: detects `_bwComponent` flag, calls `handle.mount(targetEl)` instead. | **6 hidden behaviors under 3 letters.** v1.x `bw.DOM()` was "select + optionally set innerHTML, return array." v2.x is a mount-point lifecycle manager. Same name, different contract. |
| 8 | **Hooks fire** | Split between `createDOM()` (rAF-deferred `o.mounted`/`o.render`) and `handle.mount()` (synchronous: `willMount` → create → collect refs → resolve bindings → `mounted` → `o.render`). | Two different timing models for the same concept. `createDOM` defers via rAF; `handle.mount()` is synchronous. |
| 9 | **Runtime** | **4 competing patterns, no guidance**: (a) `bw.patch(id, val)` — targeted text/attr update. (b) `bw.update(el)` — calls `_bw_render`, emits `bw:statechange`. (c) `bw.DOM('#x', newTaco)` — full teardown+rebuild. (d) `handle.set({k:v})` — reactive bindings, scheduled microtask flush. | Developer must choose between 4 patterns. Each has different perf, different side effects. No documentation on when to use which. `patch` and `update` are for raw TACO path; `set` is for ComponentHandle path; `bw.DOM` works for both but is the most expensive. |
| 10 | **Destruction** | `bw.cleanup(el)` walks `[data-bw_id]` descendants, calls unmount callbacks, removes subs, deregisters. `handle.destroy()` adds action deregistration + parent/child cleanup. | Mostly clean. Two paths (cleanup vs destroy) is inherent to having two component models. |

### Where confusion lives (summary)

**Stage 4 ↔ Stage 7 coupling.** `loadDefaultStyles()` must be called before mounting
for components to look right, and it mutates global state (body class, `bw.u`,
internal vars). Styles and mounting should be independent.

**Stages 5+6+8 collapsed.** `createDOM()` fuses element creation, node registration,
and hook firing into one call. You can't create without registering, can't register
without potentially firing hooks.

**Stage 7 overloaded.** `bw.DOM()` does too much: cleanup + state save/restore +
clear + create + mount + ComponentHandle detection. In v1.x it was "select and
optionally set innerHTML."

**Stage 9 fragmented.** Four update patterns with no clear guidance. This is
partially by design (different levels of reactivity) but needs explicit documentation
of when to use each pattern.

### Key principle: stages are independent

Styles don't depend on component lifecycle. Components don't depend on styles.
You can:
- Create components with no styles loaded (structural CSS only — functional but plain)
- Load styles with no components on the page
- Load styles, create components, change styles later
- Have multiple style scopes with different themes on one page
- Use your own CSS with bitwrench components
- Use bitwrench styles with your own HTML

The current code violates this: `loadDefaultStyles()` must be called before
`bw.DOM()` for components to look right, and it has lifecycle-like side effects
that entangle it with the component system.

### Recommendations

These are ordered by impact and risk. R1 is the styles cleanup (main topic of
this doc). R2-R4 address the other lifecycle problems exposed by the audit.

#### R1: Decouple styles (Stage 4) — v2.0.18

Already described in this doc. `makeStyles()` pure, `applyStyles()` injects one
`<style>`, `loadStyles()` is sugar. No body class, no internal state mutations.
`bw.u` removed from core entirely (see §`bw.utilCSS`).
Components render correctly with or without styles loaded.

This is the highest-impact, lowest-risk change. It fixes the class of bugs
that cost us a full debugging session (the nav-link specificity war).

#### R2: Stop firing hooks inside `createDOM()` — v2.0.18 or v2.0.19

**Problem:** `createDOM()` collapses stages 5+6+8. It creates DOM, registers
nodes, AND fires `o.mounted`/`o.render` via rAF. This means you can't create
a DOM tree without also scheduling hook execution.

**Recommendation:** `createDOM()` should create elements, set attributes, attach
event listeners, register nodes, and return. That's it. No hooks.

Hooks fire in the **mounting** function (`bw.DOM`), after the tree is in the
document. This is when hooks actually make sense — you can measure layout, query
siblings, start animations.

```
// createDOM: stages 5+6 only
var el = bw.createDOM(taco);   // creates DOM tree, registers nodes
// el is detached — no hooks have fired

// bw.DOM: stages 7+8
bw.DOM('#app', el);            // mounts, THEN fires hooks
```

**Migration risk:** Low. The rAF deferral in `createDOM()` already handles the
"not yet in document" case — we're just making it explicit. The only breakage
would be code that calls `bw.createDOM()` without ever mounting (hooks would
never fire, which is correct — why fire `mounted` on something that never mounted?).

For the ComponentHandle path, `handle.mount()` already fires hooks after
appendChild. No change needed there.

#### R3: Clarify `bw.DOM()` contract — v2.0.18

**Problem:** `bw.DOM()` does 6 things under 3 letters. In v1.x it was "select +
optionally set innerHTML, return array."

**Recommendation:** Accept that v2.x `bw.DOM()` IS the mount function. Don't
pretend it's a selector. Rename the concept in our heads:

```
bw.DOM(target, content)    // "mount content into target"
```

It does:
1. Find target element (via `bw._el()`)
2. Clean up existing children (`bw.cleanup()`)
3. Clear target's innerHTML
4. Create DOM from content (`bw.createDOM()` or `handle.mount()`)
5. Append to target
6. Fire hooks (per R2, moved here from createDOM)

That's a mount operation. Steps 1-6 are the correct sequence for mounting.
The issue isn't that it does too much — it's that the name suggests something
simpler than what it is.

**Don't rename it.** `bw.DOM()` is already established. But document it as
"mount" and stop using it as a selector. Selection is `bw.$()` (which is fine
as a convenience — no jQuery baggage, just `querySelectorAll` returning an array).

The state save/restore on the target element (saving `_bw_state`, `_bw_render`,
`_bw_subs` before cleanup, restoring after) — this IS correct behavior. The target
is the mount point; its own state survives content replacement. But it should be
documented, not hidden.

#### R4: Document the update patterns — v2.0.18 docs

**Problem:** 4 update patterns, no guidance.

**Recommendation:** Don't reduce to one pattern. They exist for different use
cases. But document clearly when to use each:

| Pattern | Use when | Cost | Example |
|---------|----------|------|---------|
| `bw.patch(id, val)` | Changing a single value (counter, status text, badge) | Cheapest — one DOM property write | `bw.patch('count', state.n)` |
| `bw.update(el)` | Re-rendering a section (dashboard panel, list after data change) | Medium — calls `o.render`, rebuilds subtree | `state.items.push(x); bw.update(listEl)` |
| `bw.DOM('#x', taco)` | Replacing entire content (page navigation, tab switch) | Expensive — full teardown + rebuild | `bw.DOM('#main', buildPage(route))` |
| `handle.set({k:v})` | Reactive component with bindings | Cheapest for bound values — only bound nodes update | `counter.set({ count: n })` |

**Rule of thumb:** Use the cheapest pattern that does the job.
- One value changing? → `patch`
- Multiple values or structure change? → `update` or `DOM`
- Using ComponentHandle? → `set`

This isn't a code change — it's a documentation change. Add to the State
Management docs page and the Quick Start guide.

#### R5: Consider — should `o.mounted` be deprecated? (discuss, not v2.0.18)

`o.mounted` and `o.render` compete at stage 3. If you provide both, `o.render`
wins (with a warning). They serve similar purposes:
- `o.mounted(el, state)` — run code after mount. Legacy pattern. One-shot.
- `o.render(el, state)` — run code after mount AND on every `bw.update()` call.

`o.render` is strictly more capable. `o.mounted` exists for backward compat.
Long-term, deprecate `o.mounted` in favor of `o.render` with an `initialRender`
flag, or keep both but document that `o.render` is preferred.

Not urgent. No code change for v2.0.18. Just a note for future.

## Resolved Design Questions

Answers to open questions raised during review.

**`applyStyles()` return value:** Returns the `<style>` DOM element. The element
has a trackable id (e.g. `bw_style_my_dashboard`) readable via `.id`.
To remove it later, pass the same scope: `bw.clearStyles('#my-dashboard')`.
Multiple `applyStyles()` calls with different scopes create separate `<style>`
elements — developer decides which to keep or remove.

**`clearStyles()` parameter:** Takes the same scope you passed to `applyStyles()`,
or a bare name for built-in styles. Conversion rule: strip `#` and `.` prefixes,
replace `-` with `_`, prepend `bw_style_`. Result is used as the `id` to find
the `<style>` element.
Examples: `'#my-dashboard'` → `bw_style_my_dashboard`. `'.preview'` → `bw_style_preview`.
`'reset'` → `bw_style_reset`. No arg → `bw_style_global`.

**Zero-config `loadStyles()`:** `loadStyles()` with no args uses the bitwrench
default palette (`#006666` teal + `#cc6633` secondary). Equivalent to
`applyStyles(makeStyles({ primary: '#006666', secondary: '#cc6633' }))`.
This is the "just make it look good" path for embedded devs.

**`bw.u` removed from core:** The entire static dictionary (54 entries, 80 lines)
is removed from `src/bitwrench.js`. Problems: hardcoded color entries (`bgTeal`)
mutated by `generateTheme()`; inline styles fight theme classes; 54 named entries
where a rule engine generates hundreds. Replaced by `bw.utilCSS` — a rule-based
parser in a separate plugin file. See §`bw.utilCSS` Plugin.
Palette colors accessed via `styles.palette.primary.base` — explicit, not magic.

**Named themes:** No first-class `bw.themePresets` registry. Presets are just
example configs in the docs. Making themes is easy enough that a registry adds
complexity without value. The docs show:
```javascript
// Ocean theme — just a config object
bw.loadStyles({ primary: '#0369a1', secondary: '#0891b2', spacing: 'normal', radius: 'md' });
```

**`makeStyles()` full config params:** All optional. Defaults to bitwrench palette.
```javascript
bw.makeStyles({
  primary:    '#4f46e5',      // default '#006666' → .bw_primary, .bw_btn_primary, .bw_bg_primary, etc.
  secondary:  '#d97706',      // default '#cc6633' → .bw_secondary, .bw_btn_secondary, .bw_bg_secondary, etc.
  tertiary:   '#10b981',      // → .bw_tertiary, .bw_btn_tertiary, .bw_bg_tertiary, etc.
  background: '#fafafe',      // → body/page background tint (hue-matched to primary)
  surface:    '#f5f3ff',      // → .bw_card background, panel backgrounds
  spacing:    'normal',       // → padding/margin tokens: compact(0.5rem) | normal(1rem) | spacious(1.5rem)
  radius:     'md',           // → border-radius: none(0) | sm(0.25rem) | md(0.375rem) | lg(0.5rem) | pill(50rem)
  globals:    false,          // → if true, also emit bare a, h1-h6, body selectors
  prefix:     'bw_'           // → class prefix (reserved, not yet wired)
});
```

**Multiple applies without scope:** Two calls to `applyStyles(stylesA)` then
`applyStyles(stylesB)` both without scope → B replaces A (same id
`bw_style_global`, idempotent). Two scoped calls to different selectors create
separate `<style>` elements with separate ids. The developer tracks by scope
and decides what stays.

## Will This Reduce Inline Styles on Pages?

Short answer: yes, significantly. But not to zero — and that's OK.

### What `pages/index.html` looks like now

833 lines. **~220 lines of `bw.injectCSS(bw.css({...}))` hand-written CSS** at
the top (lines 25-300). **35 inline `style:` attributes** scattered throughout
the TACO objects. Many are ad-hoc: `font-size: 0.8125rem`, `margin: 0 0 0.75rem`,
`color: var(--bw_teal)`, `display: flex; gap: 0.5rem`.

### Why this happens

Three causes:

1. **No BCCL component for the job.** The hero section, install strip, stat grid,
   how-it-works layout — these are page-specific compositions, not reusable
   components. They need custom CSS. That's legitimate.

2. **BCCL components exist but don't cover the styling need.** `makeCard` exists
   but the page wants a specific card variant with custom spacing, custom color.
   The developer reaches for inline `style:` because the theme system doesn't
   expose enough design tokens.

3. **Spacing and typography tweaks.** `margin: 0 0 0.75rem`, `font-size: 0.8125rem`,
   `font-weight: 600` — these should come from utility classes, not inline styles.

### What the styles cleanup fixes

**Cause 2:** `makeStyles()` returns a full palette. Instead of `color: var(--bw_teal)`
or `color: '#006666'` inline, the developer writes:
```javascript
{ t: 'h3', a: { class: 'bw_text_primary' }, c: 'Title' }
```
The themed CSS has `.bw_text_primary { color: #4f46e5; }` — no inline style needed.

**Cause 3:** The structural/styled split means utility classes cover common
spacing and typography:
```javascript
// Instead of: style: 'margin: 0 0 0.75rem; font-size: 0.8125rem; font-weight: 600'
// Use: class: 'bw_mb_3 bw_fs_sm bw_fw_semibold'
```
These utility classes are emitted by `makeStyles()` as part of the themed CSS.

### What it doesn't fix

**Cause 1:** Page-specific layouts (hero gradients, stat grids, install strips)
still need custom CSS. But they should use `bw.css()` with palette colors:
```javascript
var styles = bw.makeStyles({ primary: '#4f46e5' });
bw.injectCSS(bw.css({
  '.bw_hero_landing': {
    background: 'linear-gradient(135deg, ' + styles.palette.primary.base + ' 0%, ' + styles.palette.primary.active + ' 100%)',
    color: styles.palette.primary.textOn,
    padding: '1.25rem 0 2rem'
  }
}));
```
No more hard-coded `#006666` — the hero uses the same palette as everything else.

### Target for `pages/index.html`

| Now | After cleanup |
|-----|--------------|
| 220 lines hand-written CSS | ~80 lines (page-specific layouts only, using palette) |
| 35 inline styles | ~10 (truly one-off positioning, nothing that could be a class) |
| Hard-coded `#006666`, `#003d3d` | Zero — all from `styles.palette.*` |
| `var(--bw_teal)`, `var(--bw_text)` | Zero — replaced by `bw_text_primary`, `bw_text_muted` classes |

### The principle

Inline `style:` should be used only for values that are:
- Truly one-off (this specific element's position)
- Dynamic (computed at runtime, can't be a class)
- Structural (part of the component's layout, baked into the template)

Everything else — colors, typography, spacing, borders — belongs in classes
targeted by the theme system.

## `bw.utilCSS` Plugin — Rule-Based Utility CSS

**Status:** Separate plugin (like `bitwrench-code-edit`). Not in core bundle.

### What it replaces

The old `bw.u` was a static dictionary of 54 inline style objects baked into core
bitwrench. Problems:
- 80 lines shipped in every build
- 5 hardcoded color entries (`bgTeal`, `textTeal`, etc.) mutated by `generateTheme()`
- Naming baked in specific colors (`bgTeal` that secretly means "whatever primary is")
- 54 named entries where a rule engine generates hundreds
- Duplicated what CSS utility classes already provide once styles are loaded

### The new model

A **parser** that takes space-separated shorthand strings and expands them using
**rules**, not a lookup table. A handful of rules generate the full spacing,
typography, layout, and flex utility space.

### API

Three static methods on the function. Each takes the same shorthand string,
returns a different output format:

```javascript
bw.u('flex gap4 p4')        // → { display: 'flex', gap: '1rem', padding: '1rem' }
bw.u.css('flex gap4 p4')    // → "display:flex;gap:1rem;padding:1rem"
bw.u.cls('flex gap4 p4')    // → "bw_flex bw_gap_4 bw_p_4"
```

| Method | Returns | Use in TACO |
|--------|---------|-------------|
| `bw.u(str)` | Plain style object | `a: { style: bw.u('flex gap4 p4') }` |
| `bw.u.css(str)` | CSS style string | `a: { style: bw.u.css('flex gap4 p4') }` |
| `bw.u.cls(str)` | Space-separated class names (prefix: `bw_`) | `a: { class: bw.u.cls('flex gap4 p4') }` |

Default call returns a plain object because that's what TACO `a: { style: {...} }`
expects. No wrapper objects, no ambiguity. The `.cls()` output emits `bw_`-prefixed
class names matching what `loadStyles()` / `makeStyles()` emit.

### Rule engine

Instead of listing every variant as a static entry, compact rules generate the
full utility space:

```javascript
// One rule covers p0 through p12
['p(n)',  function(n) { return { padding: (n * 0.25) + 'rem' }; }]
// Plus directional variants
['pt(n)', function(n) { return { paddingTop: (n * 0.25) + 'rem' }; }]
['px(n)', function(n) { return { paddingLeft: (n*0.25)+'rem', paddingRight: (n*0.25)+'rem' }; }]
// Same pattern for margin, gap
['m(n)',   function(n) { return { margin: (n * 0.25) + 'rem' }; }]
['gap(n)', function(n) { return { gap: (n * 0.25) + 'rem' }; }]
```

Static keywords (no numeric param) map directly:

```javascript
'flex'    → { display: 'flex' }
'flexCol' → { display: 'flex', flexDirection: 'column' }
'hidden'  → { display: 'none' }
'bold'    → { fontWeight: '700' }
```

A handful of rules generate hundreds of utilities. The v1 `h1..h6` internal rules
used the same pattern — a rule, not 6 separate entries.

### Extensibility

Users register custom rules via `.extend()`:

```javascript
bw.u.extend({
  'brand':    { background: '#4f46e5', color: '#fff' },
  'icon-(n)': function(n) { return { width: n + 'px', height: n + 'px' }; }
});

bw.u('flex brand icon-24')
// → { display: 'flex', background: '#4f46e5', color: '#fff', width: '24px', height: '24px' }
```

### Colors

Minimal. Layout/typography/spacing only by default. For the no-CSS embedded case,
support named CSS colors and `[#hex]` escape hatch:

```javascript
bw.u('bg-white text-black')     // → { background: 'white', color: 'black' }
bw.u('bg-[#4f46e5]')            // → { background: '#4f46e5' }
```

No palette coupling. No `bgTeal` that secretly means "primary." If you write a hex,
you get that hex. Palette colors belong in CSS classes (`bw_text_primary`, etc.),
not inline styles.

### When to use

**Use CSS classes when you can, use `bw.u()` when you can't.** Scenarios where
`bw.u()` earns its keep:

- Embedded devices with no CSS loaded (no `<style>` element at all)
- Email templates where `<style>` tags are stripped
- Dynamic values that can't be predetermined classes
- Self-contained TACOs that must work without any stylesheet dependency

When styles are loaded, prefer classes:
```javascript
// Preferred (themed, responsive, overridable):
{ t: 'div', a: { class: 'bw_flex bw_gap_4 bw_p_4' }, c: '...' }

// bw.u — for no-CSS scenarios:
{ t: 'div', a: { style: bw.u('flex gap4 p4') }, c: '...' }
```

### Plugin architecture

Separate source file: `src/bitwrench-util-css.js`. Built as its own dist artifact
(like `bitwrench-code-edit`). Not bundled into core.

```html
<script src="bitwrench.umd.js"></script>
<script src="bitwrench-util-css.umd.js"></script>  <!-- only if you need it -->
```

Attaches as `bw.utilCSS` with alias `bw.u`. Core bitwrench ships without it —
zero impact on the 45KB budget.

### Relationship to `bw.s()`

`bw.s()` stays in core (6 lines). It's a general-purpose style object merge:
```javascript
bw.s(baseStyle, isActive && activeOverride, { color: 'red' })
```
Useful with or without `bw.utilCSS`. When `bw.u()` returns objects, `bw.s()` can
merge them with ad-hoc overrides. When `bw.u.css()` returns strings, `bw.s()` is
not needed (string concatenation works). Both coexist naturally.

## Open Questions

1. **Naming:** ~~`makeStyles/applyStyles/loadStyles` vs `makeTheme/applyTheme/loadTheme`?~~
   **Resolved: `makeStyles`.** Consistent with `makeCard`, `makeButton` pattern.
   "make" = create data from config. "styles" is broader than "theme" and accurate.

2. **Auto-load:** ~~Should `bw.DOM()` auto-call `loadStyles()` on first use if no styles
   are loaded?~~ **No.** Violates the no-magic principle. If you want styles, call
   `loadStyles()`. Components without styles = functional but unstyled. That's fine.

3. **CSS custom properties:** Should scoped themes use `--bw_*` variables? This would
   make theming much simpler (just override variables) but breaks IE11 story.
   Counter-argument: IE11 is effectively dead. Do we still care?

4. **Bootstrap-compat utilities:** The `.fs-*`, `.fw-*`, etc. selectors without `bw_`
   prefix are global namespace pollution. Keep for migration ease, or drop and use
   only `bw_` prefixed versions?

5. **`[class*="col-"]` attribute selector:** This matches any class containing "col-".
   Replace with explicit `.bw_col_*` selectors?

6. **Prefix implementation complexity:** ~~Worth it for v2.0.18 or defer to v2.1.0?~~
   **Resolved: punt to v2.1.0.** API shape ready (`prefix` field in config), plumbing
   not wired. Too invasive for this release. Developers who need it today do string
   replacement on the CSS output.

7. **Structural spacing defaults:** Which components need built-in structural padding?
   Candidates: `bw_btn` (must be clickable), `bw_card_body` (content not flush),
   `bw_table td/th` (readable). Use `em`-based units so they scale with context.
   Themed spacing overrides these.

8. **Component registry granularity:** Should `bw.components` store full TACO trees
   (with nested children) or just the root element with slots? Full trees are more
   inspectable; slot-based is more flexible. Leaning full trees — it's what the dev
   sees in the output, so it's what they should see in the registry.

9. **Structural CSS delivery:** ~~Three options: A (in makeStyles), B (separate fn), C (inline on templates).~~
   **Resolved: A+C hybrid.** Fixed layout properties that define what a component IS
   go inline on the template (`style: { display: 'flex', cursor: 'pointer' }`).
   Structural properties that span many components (e.g. `box-sizing: border-box`
   on all `bw_*`, `border-collapse` on tables) go in `makeStyles()` output.
   No separate `makeStructuralStyles()` function needed.
