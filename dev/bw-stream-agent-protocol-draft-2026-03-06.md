
# bw-stream: A Minimal Protocol for Agent-Driven and Server-Driven UI

## Status

**Proposed Architecture (Exploratory)**

This document proposes a minimal streaming UI protocol built around the Bitwrench (bw) rendering model. The goal is to support:

- agent-driven UI
- server-driven UI
- embedded device UI
- streaming UI updates
- safe declarative rendering

The proposal intentionally keeps the protocol **small and renderer-centric**, allowing policy and governance layers to exist outside the core runtime.

This design is motivated in part by examining newer agent UI protocols such as A2UI and AG-UI. The argument of this document is that a **minimal mutation protocol plus a structured UI object model** can cover most practical use cases while remaining significantly simpler.

This document is intentionally explicit and reflective so that future development decisions can be evaluated against the original reasoning.



# Design Goals

The bw-stream protocol should satisfy several goals.

## Simplicity

The protocol must remain extremely small and easy to implement.

The renderer should not require:

- large lifecycle systems
- component registries
- reconciliation engines
- compilers

Bitwrench already provides a renderer that maps object structures directly to DOM nodes.

The protocol should therefore **focus on mutation messages rather than describing rendering engines**.



## Declarative UI

Messages must describe **UI structure and changes**, not imperative UI code.

This allows UI to be safely generated by:

- servers
- AI agents
- embedded devices

Because messages are structured JSON, they remain inspectable and policy-filterable.



## Streaming Updates

Modern interactive systems frequently require incremental UI updates.

Examples include:

- chat interfaces
- progressive computation output
- telemetry dashboards
- log streams
- notebook-like environments

The protocol must support streaming without requiring full page re-renders.



## Renderer Independence

The protocol should not assume a specific rendering engine beyond the existence of a UI tree.

Bitwrench is currently the target renderer, but the protocol design should remain conceptually separable from any particular runtime.



# Core Concept

The bw-stream protocol separates three layers that many other UI protocols mix together:

```

transport  
policy  
renderer  

```

The protocol defines **how UI updates are transmitted**, not how they are governed or implemented.



# Core Protocol Concepts

The protocol is built around three primitives:

1. **surfaces**
2. **mutation operations**
3. **UI objects**

These primitives together allow a wide range of UI behaviors to be expressed.



# Surfaces

A **surface** represents a named UI region.

Surfaces are logical containers where UI elements are rendered.

Examples include:

```

main  
sidebar  
dialog  
chat  
log  
footer  

````

Surfaces provide a controlled entry point for UI rendering.

Example message defining a surface:

```json
{  
  "type": "surface",  
  "id": "main"  
}  
````

In a browser implementation, surfaces typically map to DOM elements.

Example mapping:

```
surface.main → #main  
surface.sidebar → #sidebar  
```



# Why Surfaces Matter

Surfaces provide several advantages:

* **Agent reasoning simplicity** - agents can target high-level regions rather than arbitrary selectors.

* **Safety boundary** - rendering is restricted to known regions of the UI.

* **Cross-renderer compatibility** - surfaces translate easily to other UI frameworks.

This idea aligns closely with the "surface" concept used in A2UI.



# Mutation Operations

The bw-stream protocol uses a **very small mutation vocabulary**.

The following operations are proposed.



## render

Render a new UI subtree into a surface.

Example:

```json
{  
  "op": "render",  
  "surface": "main",  
  "node": {  
    "t": "div",  
    "c": "Hello"  
  }  
}  
```



## replace

Replace an existing subtree with a new node.

Example:

```json
{  
  "op": "replace",  
  "target": "bw_uuid_9283",  
  "node": {  
    "t": "div",  
    "c": "Updated content"  
  }  
}  
```



## append

Append a node to a container.

Example:

```json
{  
  "op": "append",  
  "surface": "chat",  
  "node": {  
    "t": "div",  
    "a": {"class":"msg agent"},  
    "c": "Hello human"  
  }  
}  
```

This operation is essential for streaming interfaces.



## patch

Modify attributes or content of an existing node.

Example:

```json
{  
  "op": "patch",  
  "target": "bw_uuid_1843",  
  "attr": {  
    "class": "complete"  
  }  
}  
```



## remove

Remove a node entirely.

Example:

```json
{  
  "op": "remove",  
  "target": "bw_uuid_1843"  
}  
```



# UI Object Model

The bw-stream protocol uses the native bitwrench object representation.

Example:

```json
{  
  "t": "div",  
  "a": {"class":"card"},  
  "c": [  
    {"t":"h2","c":"Sensor"},  
    {"t":"span","c":"24.5C"}  
  ]  
}  
```

Where:

* `t` – tag
* `a` – attributes
* `c` – children
* `o` – optional metadata



# Metadata Channel

Bitwrench objects support an optional metadata field:

```
o:{}  
```

Example:

```json
{  
  "t":"div",  
  "c":"Temperature",  
  "o":{  
    "intent":"display_sensor",  
    "component":"card",  
    "confidence":0.87  
  }  
}  
```

Metadata allows additional information to accompany UI structures without affecting rendering.

Possible uses include:

* AI reasoning hints
* component suggestions
* analytics
* provenance
* policy signals

This allows semantic information to flow through the system without bloating the core protocol.



# Streaming Model

bw-stream assumes messages are delivered as a stream.

Possible transports include:

```
HTTP chunked  
Server Sent Events  
WebSockets  
IPC pipes  
stdin/stdout  
```

Each message is a self-contained JSON structure.

Example sequence:

```
render main  
append chat  
append chat  
patch spinner  
replace results  
```



# Policy Layer

The protocol intentionally **does not enforce UI governance directly**.

Instead, a policy layer can be inserted before rendering.

Example policy checks:

```
allowed tags  
allowed attributes  
allowed surfaces  
allowed components  
```

This layer can also implement component catalogs.



# Component Catalog (Optional)

Some environments may require a restricted component vocabulary.

Example catalog:

```
Card  
Button  
Text  
Form  
Table  
ChatMessage  
```

Agents can reference these through metadata.

Example:

```json
{  
  "t":"div",  
  "o":{  
    "component":"Card"  
  }  
}  
```

The renderer or policy layer can expand these into concrete UI structures.

This allows safe UI generation without forcing the catalog concept into the core protocol.



# Node Addressing

Bitwrench already assigns UUID classes to created nodes.

Example:

```
bw_uuid_ab12  
```

The runtime can maintain a registry:

```
bw.nodeRegistry[bw_uuid] → DOM node  
```

This enables constant-time lookup for mutation operations.

Selectors remain available as fallback addressing.



# Why This Protocol Is Small

The entire protocol relies on:

```
surfaces  
5 mutation operations  
UI objects  
metadata  
```

Everything else (catalogs, policies, adapters) is optional.



# Relationship to A2UI

A2UI introduces:

```
surfaces  
component catalogs  
stream updates  
safe declarative UI  
```

The bw-stream protocol can express these ideas using simpler primitives.

| A2UI Concept      | bw-stream Equivalent |
| -- | -- |
| surface           | surface              |
| updateComponents  | replace / append     |
| updateDataModel   | patch                |
| deleteSurface     | remove               |
| component catalog | policy + metadata    |



# Relationship to AG-UI

AG-UI focuses on:

* agent interaction
* UI streaming
* tool invocation
* chat interfaces

The bw-stream mutation model supports these patterns naturally.

Example chat stream:

```
append chat user message  
append chat agent message  
append chat reasoning  
append chat final response  
```



# Advantages of the bw-stream Model

The design offers several advantages.

* **Small protocol surface** – fewer rules and message types.

* **Renderer simplicity** – the renderer performs direct UI mutations rather than lifecycle orchestration.

* **Token efficiency** – LLM-generated structures are smaller than complex schemas.

* **Embedded compatibility** – microcontrollers can easily generate object trees.

* **Streaming friendliness** – append operations support progressive rendering.



# Limitations

The protocol intentionally avoids solving certain problems.

These include:

* global UI state models
* cross-renderer component semantics
* application navigation
* access control

These concerns belong in surrounding systems.



# Conclusion

The bw-stream protocol proposes a minimal, renderer-centric approach to streaming UI updates.

By combining:

```
surfaces  
small mutation vocabulary  
structured UI objects  
metadata channel  
optional policy layer  
```

the protocol can support a wide range of modern UI scenarios including:

* embedded dashboards
* server-driven interfaces
* AI-generated UI
* streaming chat systems
* progressive result rendering

The design intentionally keeps the protocol simple and extensible, allowing governance and policy to evolve independently from the rendering system.
