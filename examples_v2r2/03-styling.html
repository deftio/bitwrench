Styling

Where bitwrench styles come from, utility CSS as JavaScript objects, and dynamic theming.

The CSS Origin Story

Bitwrench includes ~22 component styles (buttons, cards, tables, forms, alerts, badges, tabs, and more). You get them two ways:

Option 1: CSS File

<link href="bitwrench.css" rel="stylesheet">

Pre-built CSS file. Include it like Bootstrap or any other stylesheet. This is what these example pages use.

Option 2: Runtime Generation

bw.loadDefaultStyles();

Generate styles from JavaScript objects at runtime. No CSS file needed — useful for embedded widgets or when you want full JS control.

Both produce the same bw-btn, bw-card, bw-table, and other classes. The difference is just delivery: static file vs. runtime injection.

Why this matters: Because bitwrench styles are JavaScript objects internally, you can compose them, compute them, override them, and send them from a server. This is the foundation for everything else on this page.

Styles as JavaScript Objects

In bitwrench, styles are plain objects. bw.s() merges them (like Object.assign), and bw.u provides pre-built utilities:

How it works

bw.u contains ~50 utility style objects (like Tailwind classes, but as JS). bw.s() merges them left-to-right, last one wins. Unlike Tailwind, there is no build step — your styles ARE your code.

Available utilities: bw.u.flex, bw.u.flexCol, bw.u.gap4, bw.u.p4, bw.u.rounded, bw.u.bold, bw.u.textCenter, bw.u.bgTeal, bw.u.wFull, bw.u.transition, and more.

bw.css() — Generate CSS Classes from Objects

For pseudo-classes (:hover, :focus) and media queries, you need actual CSS classes. bw.css() generates CSS strings from JS objects, and bw.injectCSS() adds them to the page:

Key insight: bw.css() turns { '.my-btn': { background: 'teal' } } into the CSS string .my-btn { background: teal; }. You write styles as data, bitwrench handles the syntax.

Responsive Styles

bw.responsive() generates mobile-first media query CSS from a single object. Breakpoints: sm (640px), md (768px), lg (1024px), xl (1280px).

Inline vs Classes — When to Use Which

Inline Styles (bw.s + bw.u)

  • Simple, no name collisions
  • Great for computed/dynamic styles
  • Work with TACO a: { style: ... }
  • Can be sent from a server as JSON

Use for: one-off styles, runtime-computed values, server-driven UI

CSS Classes (bw.css + bw.injectCSS)

  • Required for :hover, :focus, :active
  • Required for media queries / responsiveness
  • Reusable across many elements
  • Can be inspected in DevTools by name

Use for: pseudo-classes, responsive layouts, shared component styles

Honest trade-off: Bitwrench's CSS approach is genuinely powerful — styles are composable JS objects, computable at runtime, and sendable from any backend. What it cannot match: Tailwind's IDE autocomplete, tree-shaking, and community ecosystem.
