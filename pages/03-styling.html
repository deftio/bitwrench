Styling

Where bitwrench styles come from, utility CSS as JavaScript objects, and dynamic theming.

Three Ways to Style Bitwrench Elements

Bitwrench generates DOM elements with class and style attributes. Whatever CSS you use — your own stylesheet, Bootstrap, Tailwind, bitwrench.css, or no CSS file at all — it just works. You can mix all three approaches on the same element:

1. Inline Styles

No CSS file needed. Use bw.s() and bw.u to compose style objects directly on elements. JavaScript IS your style system.

a: { style: bw.s(bw.u.flex, bw.u.p4) }

2. Generated Classes

Use bw.css() to create CSS rules from JS objects when you need :hover, :focus, or media queries.

bw.injectCSS(bw.css({
  '.card:hover': { ... }
}))

3. Any CSS File

Bring your own classes. bitwrench.css is a convenience starter kit — or use Bootstrap, Tailwind, your own stylesheet, or nothing.

a: { class: 'my-class' }
History note: In bitwrench 1.x, the library auto-generated its CSS at runtime if it detected no stylesheet was loaded. In 2.x, this is more à la carte — you choose what to load and when. The DNA has always been "CSS from JS."

Inline Styles — No CSS File Needed

bw.s() merges style objects (like Object.assign). bw.u provides ~50 pre-built utilities. Together they replace CSS variables, preprocessors, and utility class frameworks — because JavaScript variables ARE your variables.

Why CSS variables aren't needed

CSS variables (:root { --accent: teal }) exist to share values between rules. But in bitwrench, JS variables already do that — var accent = '#006666' and you use it in inline styles, generated classes, or both. The ? : ternary operator replaces @media and conditional class toggling. No :root declarations needed.

Computed Classes — Classes Are Just Strings

Class names in bitwrench are just strings in the a: { class: ... } attribute. You can compute them with JavaScript — concatenation, ternary operators, .map() — no special directive syntax needed. The classes can come from bitwrench.css, your own stylesheet, Bootstrap, Tailwind, or anything.

Bring any CSS framework

Since bitwrench just sets the class attribute, you can use any CSS classes: 'btn btn-primary' (Bootstrap), 'flex gap-4 p-4' (Tailwind), or your own. You could even use bitwrench's bw.css() to generate Tailwind-equivalent styles at runtime — turning class name strings into real CSS without Tailwind's build pipeline.

Generated Classes — When You Need :hover or Media Queries

Inline styles can't express :hover, :focus, or @media. For those, generate CSS at runtime with bw.css() + bw.injectCSS():

Key insight: bw.css() turns { '.my-btn': { background: 'teal' } } into the CSS string .my-btn { background: teal; }. You write styles as data, bitwrench handles the syntax.

Responsive Styles

bw.responsive() generates mobile-first media query CSS from a single object. Breakpoints: sm (640px), md (768px), lg (1024px), xl (1280px).

Mixing All Three on One Element

The real power: use a global class for base layout, a generated class for hover behavior, and an inline style for the dynamic part. All on the same element.

Global Styles — The Convenience Option

Bitwrench ships with bitwrench.css — ~22 component styles (buttons, cards, tables, forms, alerts, badges, tabs). It's a convenience starter kit, not a requirement. You get them two ways:

Static CSS file

<link href="bitwrench.css" rel="stylesheet">

Include it like any stylesheet. These example pages use this approach.

Runtime generation

bw.loadDefaultStyles();

Generate the same styles from JS at runtime. Useful for embedded widgets or single-file deployments.

Both produce the same bw-btn, bw-card, bw-table classes. Or skip them entirely and use your own CSS, Bootstrap, Tailwind, or pure inline styles.
