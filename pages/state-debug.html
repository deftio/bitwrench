State Management Test

Independent component state, getter/setter access, and cross-component coordination.

What This Page Tests

This page validates bitwrench's state management architecture with concrete, interactive tests. Three counter cards are created from the same factory function. Each card has local buttons that modify only its own state. A global control panel can read and modify all three cards, proving that external access via getters and setters works correctly.

New API patterns demonstrated

o.render — first-class render function stored as el._bw_render. Replaces the o.mounted boilerplate pattern. Called automatically on mount.
bw.update(el) — re-runs el._bw_render. Replaces manual render(el) calls. Emits bw:statechange so other components can react.
bw.patch(id, value) — targeted DOM update by UUID. Changes one value without rebuilding the component tree.
bw.emit() / bw.on() — lightweight pub/sub via CustomEvent. Cards emit bw:statechange; global panel listens. No tight coupling.

Global Coordination

These buttons operate across all three cards using the getter/setter API returned by each factory call. The global panel auto-updates via bw.on() — cards emit bw:statechange on every state mutation.

Live State Inspector

Reads el._bw_state directly from each counter card's DOM element to verify that displayed values and internal state are always in sync. Auto-updates on every action via bw.on().

Demo B: Dynamic List & Event Log

Dynamic Counter List

A list of mini-counters that can be added and removed. Each item gets a stable UUID at creation time. bw.patch() updates individual values without re-rendering the list. bw.update() rebuilds the list when items are added or removed. UUIDs survive removal — the same ID addresses the same item regardless of its position in the list.

Event Log

Every state mutation is logged here with a timestamp, the action, the card name, and the new value. Useful for verifying the sequence of operations.

Event Paradigms: emit/on vs pub/sub

Bitwrench provides two event systems. emit/on wraps DOM CustomEvents (scoped to an element subtree, bubbles up). pub/sub is application-scoped (topic-based, decoupled from the DOM tree). Both coexist.

Comparison

Aspect emit/on pub/sub
ScopeDOM subtree (bubbles)Application-wide (topic)
Trigger1:1 with DOM actionComposite condition / logic
CleanupAutomatic (DOM GC)unsub() or element-tied
CouplingSpatial (ancestor)None (any component)
