Quick Start

From zero to working UI in 2 minutes. Code first, explanations after.

Your First Bitwrench Page

This is the complete code for a bitwrench page. One script tag, one function call: 

<!DOCTYPE html>
<html>
<head>
  <script src="https://unpkg.com/bitwrench@2/dist/bitwrench.umd.js"></script>
  <link rel="stylesheet" href="https://unpkg.com/bitwrench@2/dist/bitwrench.css">
</head>
<body>
  <div id="app"></div>
  <script>
    bw.DOM('#app', { t: 'h1', c: 'Hello bitwrench!' });
  </script>
</body>
</html>

Building Blocks

Every HTML element is a JavaScript object with a tag, attributes, and content. Build up progressively:

This is called TACO

Tag, Attributes, Content, Options. You've just used it. The name comes after you've seen the idea — it's just a plain JavaScript object that maps directly to HTML. Only t is required; everything else is optional.

Where Do Styles Come From?

Bitwrench includes built-in CSS classes (bw-btn, bw-card, etc.) that give you Bootstrap-like styling. You get them by including <link href="bitwrench.css"> in your page.

Helpers return JavaScript objects, not HTML

bw.makeButton() returns a TACO object. bw.DOM() renders it. The helper is just a shortcut for writing the object yourself. See the Styling page for the full CSS story.

Making It Interactive

When state changes, call render again. Bitwrench replaces the old content with the new. That's the entire update model.

This is the simplest pattern

A closure variable and a render function is all you need for simple components. You decide when to re-render by calling the function — every update is an explicit call you can see and step through.

There's more when you need it

For real applications, bitwrench provides a full state management toolkit:

  • Component state — o.state and o.render store state and render logic directly on the DOM element. Multiple instances each carry their own independent state.
  • Targeted updates — bw.patch(id, value) updates a single element by UUID without rebuilding the component tree. Update one value in a list of 1,000 items without touching the rest.
  • Pub/sub — bw.pub() / bw.sub() provide decoupled cross-component messaging, independent of DOM hierarchy. Subscriptions can be tied to element lifecycle for automatic cleanup.
  • DOM events — bw.emit() / bw.on() wrap CustomEvents for scoped, bubbling communication between parent and child components.
  • Lifecycle — o.mounted(el) runs after the element is in the DOM. o.unmount(el) runs on cleanup for timers, subscriptions, or anything that needs teardown.

When bw.DOM() replaces content, old DOM nodes and their event listeners are removed by the browser's garbage collector — no manual listener cleanup needed. State lives on the element, so there's nothing external to leak.

Debugging: all state is visible in the browser inspector. Select any bitwrench component, then check $0._bw_state in the console to see its current state. No special devtools extension required.

See the State & Interactivity page for the full picture, including working examples of each pattern.
